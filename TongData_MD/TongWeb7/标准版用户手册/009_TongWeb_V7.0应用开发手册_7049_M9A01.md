1.1 Servlet 
Servlet是基于Java技术的Web组件，由容器管理并产生动态的内容。Servlet引擎作为Web服务器的扩展
提供支持Servlet的功能。Servlet与客户端通过Servlet容器实现的请求/响应模型进行交互。具体开发过程中使
用的Java Servlet API可参考Oracle公司提供的相关文档。TongWeb目前应用服务器支持Servlet3.1规范。 
本章节中所有提到“参考TongWeb样例”的地方，都是指位于
${TongWeb_HOME}/samples/servletjsp-samples/servlet3Jsp22-new-feature目录下的应用
Servlet3Jsp22Sample.ear，可将其部署到TongWeb之后，访问其各个子模块，了解相应的特性及用法。

异步处理的支持 
Servlet接收到请求之后，可能会对数据进行一些预处理，之后Servlet线程将请求转交给一个异步线
程来执行业务处理，线程本身返回，此时Servlet还没有生成响应数据。异步线程处理完业务以后，
可以直接生成响应数据（异步线程拥有ServletRequest和ServletResponse对象的引用），或者将请求
继续转发给其它Servlet。这样Servlet线程不再是一直处于阻塞状态以等待业务逻辑的处理，而是启

1.1.6.1 编写Filter类 
编写过滤器的API是javax.servlet包中Filter、FilterChain和FilterConfig接口中定义的一些方法。定义一个
过滤器需要实现Filter接口。Filter接口中最主要的方法是doFilter()方法，它接收三个参数：Request对象、
Response对象、FilterChain对象。 
void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) 
doFilter方法能够执行的动作包括： 
⚫ 
检查请求的头信息。 
⚫ 
定制请求对象，改变请求的头信息或数据。 
⚫ 
定制响应对象，改变响应的头信息或数据。 
⚫ 
调用在过滤器链中的下一个实体。如果当前过滤器是链中的最后一个过滤器，那么下一个实体就
是客户请求的资源，否则，链中的下一个过滤器会被调用。  
⚫ 
检查响应的头信息。 
⚫ 
抛出异常显示处理过程中的错误。 
Filter为了改变响应，必须在它被返回客户端之前截获它。但是需要注意的是，在Filter链中的任何Filter
都有可能关闭响应，结果导致响应立刻返回。Filter是串栈式的结构，所设置对响应的改变是在doFilter()函
数调用之后，也就是栈底。可是响应对象还没有到栈底就被返回了怎么办？为此，必须传递给doFilter函数
一个替身。怎么才能找到这个替身？需要创建一个响应的包装类，并且重载它的getWriter和getOutputStream
函数，以返回这个替代的流。 
标准的Servlet实现提供了现成的包装实现类：ServletRequestWrapper和HttpServletRequestWrapper，
ServletResponseWrapper和HttpServletResponseWrapper。 用户只需实现这四个中的一个，并覆盖响应的函数
即可。下面的例子就是对响应的封装，它使得以后的Filter不能直接提交响应。 
public class CharResponseWrapper extends 
   HttpServletResponseWrapper { 
   private CharArrayWriter output; 
   public String toString() { 
      return output.toString(); 
   } 
   public CharResponseWrapper(HttpServletResponse response){ 
      super(response); 
      output = new CharArrayWriter(); 
   } 
   public PrintWriter getWriter(){ 
      return new PrintWriter(output); 
   } 
}

JavaServer Pages (JSP)技术允许开发人员轻松的创建静态或动态的Web内容。JSP不仅具有像Servlet一样

1.3.2.1 恢复视图 
当请求JSF页面时，如点击按钮或链接，JSF开始恢复视图阶段。 
在这个阶段JSF建立页面的视图，给视图中的组件设置事件处理器、校验器，在FacesContext中保存视
图。FacesContext含有所有处理请求的信息，所以页面元素包括组件标签、事件处理器、转换器、校验器
都要接触FacesContext。如果该请求是第一次的请求，JSF在这个阶段产生一个空的视图，生命周期进入显
示应答阶段，这个空的视图会在页面返回的时候用到。如果请求不是第一次，对应于这个页面的视图已经
存在，JSF用存在客户端或服务器端的信息恢复视图。 
1.3.2.2 应用请求值 
在这个阶段，使用请求中的当前值来更新每个组件的当地值。这些值可能来自请求参数，请求报头及
Cookies等。在这个阶段的处理中，组件可以往事件队列里加入某些事件，这些事件可在随后的处理阶段处
理。 
可编辑值的输入控件都有个immediate属性。如果为true，将立刻进行验证，而不是要等到处理验证阶
段。按钮或者超链接也有immediate属性，如果该属性被设置为true，它将在本阶段触发动作事件。 
阶段执行后，事件会广播给激活的事件监听器。任何响应此生命周期事件的代码都可以截断生命周期，
直接跳到显示响应阶段，或者终止处理。 
1.3.2.3 处理验证 
当每个组件的本地值被更新后，在此阶段中，每个组件都有一些值需要根据应用程序的验证规则进行
有效性验证。这些验证规则可以是预先进行定义的（JSF中提供的），也可以由开发者进行定义。 
用户所输入的值会与这些验证规则进行比较。如果验证成功，将到生命周期的下一个阶段。否则控制
将跳到显示响应阶段，显示验证和转换错误信息。 
1.3.2.4 更新模型值 
只有当组件树中所有组件的本地值都通过验证后，才能有可能到达该阶段，开始处理相关的后台Bean
或模型对象。完成此阶段时，JSF将向激活的监听器广播事件。同样监听器可以跳转到呈现相应阶段。 
到这一步为止，基于用户的输入JSF更新了组件的值，验证了这些值，更新了相关的bean，但是还没有
编写任何代码。JSF将自动完成大量的UI处理任务。

2.2.3 拦截器 
拦截器可以监听程序的一个或所有方法。拦截器对方法调用流提供了细粒度控制，可以在无状态会话 
bean及有状态会话bean上使用它们。 
⚫ 
拦截器可以是同一Bean 类中的方法或者是一个外部类。  
⚫ 
拦截器可以拦截Session Bean 的方法调用或生命周期事件。  
⚫ 
拦截器按照使用范围和定义可以分为类级，方法级，和Bean 级，其中类级和方法级的拦截器的实现是
外部类。 
 
类级用标注@Interceptors 定义在Bean 的类名前，作用范围是Bean 类的所有方法。 
 
方法级用标注@ Interceptors 定义在Bean 类的某一方法前，作用范围是该方法。 
 
Bean 级的拦截器实现是Bean 类的某一方法( 非业务方法)，在该方法前使用标注@AroundInvoke
表示其为用做拦截器的方法，作用范围是除了拦截器方法自身之外Bean 类的所有方法。  
 
Bean 级别定义的拦截器方法只允许一个。类级别和方法级别的拦截器允许有多个。 
 
以上使用标注的情况均可在部署描述符中定义。 
⚫ 
拦截器的定义可以使用部署描述符和标注，部署描述符优先级高于标注，部署描述符支持通配符定义
拦截器作用范围。 
⚫ 
多个拦截器的执行顺序：定义在外部类的拦截器的执行优先级别高于定义在Bean 类中的优先级别，同
一级别的按照定义顺序执行。 
⚫ 
拦截器的实现规则：必须使用标注@AroundInvoke 表示其为用作拦截器的方法，这一方法必须遵守以下
格式：public Object xxx(javax.interceptor.InvocationContext ctx)throws Exception，其中xxx
可以任意取名，javax.interceptor.InvocationContext 封装了客户端所调用业务方法的一些信息。必
须调用InvocationContext对象的proceed()方法作为返回结果，否则其他拦截器和业务方法不会执行。 
⚫ 
拦截器的使用：如果不希望某些Bean class 使用拦截器，可以在其名称上面加入
@ExcludeDefaultInterceptors 标注。此外，拦截器中也可以使用注入。

2.2.5 Pool和Cache 
实例池(Pool) 
为提高无状态会话Bean的响应性能， EJB容器提供了实例池（Pool）机制进行Bean实例管理。 
为了避免EJB容器为用户的每次方法调用都进行实例的创建与销毁，因而影响系统性能，减少系统吞吐
量，EJB容器提供了EJB池（实例池）来应对大量用户的访问。 
当EJB应用部署到应用服务器时，EJB容器会预先创建一定数量的Bean实例放入EJB池，由于客户端应用
程序不会直接访问真正的Bean实例，而是通过EJB对象所实现的接口与Bean实例进行交互，因此EJB服务器
也就不需要为每个客户端应用分别维护相应的Bean实例，而是只需维护足以完成工作的少量Bean实例，并
重复使用这些Bean实例为不同的请求提供服务就可以了。 
如果实例池中的所有实例都在处理请求，那么EJB容器首先会判断实例池中该Bean的实例数是否已经达
到了"最大实例数"，如果没有达到这个最大实例数，EJB容器会创建新的Bean实例放入实例池中，并为客户
端服务。如果达到了"最大实例数"，那么EJB容器会挂起请求，等待有空闲的实例可用，直到超时为止。如
果池中的Bean处于空闲状态并持续一段时间，EJB容器会销毁一定数量的Bean实例。

一定时间内没有被再次访问的话，EJB容器将其状态数据保存到硬盘，并从EJB对象和缓存中删除此实例，
释放缓存，这一过程称为钝化。 
而当客户机请求此Bean时，容器会自动在缓存中创建一个新的实例，并按照钝化期间保存的内容逐一
设置数据成员，这一过程称为激活。

3.9 EntityManger的事务控制 
JPA规范中规定了实体管理器两种事务类型，JTA和RESOURCE_LOCAL。 
RESOURCE_LOCAL 事务，既能在JavaEE 环境使用，又可以在Java SE 环境中使用。代码中使用
RESOURCE_LOCAL管理事务时，要通过调用实体管理器EntityManager的getTransaction ()方法来获得本地
事务对象，进行事务的管理。 
JTA事务，只能在Java EE环境使用，事务是通过JTA方式来控制的，而不是EntityManager自己控制。
JTA事务类型的EntityManager将加入到当前事务中，事务的开始和结束都由外部程序来完成。 
EntityManager事务类型通过Persisence.xml文件Persistence-Unit中的transaction-type属性来定
义。

第4章 Dependency Inject 
DI是指Dependency Inject依赖注入，它由JSR-330（Dependency Injection for Java）定义，所在包
为javax.inject。常用标记包括@inject、@Qualifier、Provider、@Named、@Scope。 
4.1 @inject 
@Inject标识了可注入的构造器、方法或属性。它取代了传统的new一个对象实例或者从工厂方法获取
对象实例的方法，通过标记@inject来获取对象实例。可用于静态或者实例成员。 
注入顺序为构造器，字段，最后是方法。超类的字段、方法将优先于子类的字段、方法被注入。对于
同一个类的字段和方法是不区分注入顺序的。例如： 
public class Car { 
     //可注入的构造器。标记后将查找实参对构造器进行注入。 
     //构造器参数可以为空 
    @Inject public Car(Engine engine) { ... } 
     // 可注入的属性。标记后会查找Provider<Seat>并注入。 
     //非final的，属性名有效 
    @Inject private Provider<Seat> seatProvider; 
     // 可注入的方法。会查找实参并对方法注入。 
     //非abstract的，可无实参或形参，可有返回值，方法名有效 
    @Inject void install(Windshield windshield, Trunk trunk) { ... } 
   } 
当一个方法标注了@Inject并覆写了其他标注了@Inject的方法时，对于每一个实例的每一次注入请求，
该方法只会被注入一次。当一个方法没有标注@Inject并覆写了其他标注了@Inject的方法时，该方法不会
被注入。 
要进行成员注入就必须标注@Inject。一个可注入的成员可以使用任何访问修饰符（包括 private）。
不过受于平台或注入器限制（例如安全管理或缺乏反射支持），标注了@Inject的非公有成员可能将不被注
入。 
4.2 @Qualifier

第5章 CDI 
CDI是在JAVAEE6被引入的新规范，指Contexts Dependency Inject环境依赖注入，可提供一个与j2ee
环境相结合的DI的环境，实现了注入并集成生命周期管理。由JSR-299（Contexts and Dependency Injection 
for the JavaTM EE platform）定义的，所在包为javax.enterprise.context，javax.enterprise.inject，
javax.enterprise.event，javax.enterprise.util，javax.decorator等。 
CDI总的来说有以下几个方面：建立的J2EE领域的“DI”；负责针对于注入到CDI容器中Bean的生命周期
的管理；负责bean与上下文之间的联系；基于j2ee的限定符区分；bean范围管理；针对于bean的AOP的拦截
功能的interceptorBinding，是与intercepter规范进行配合的；装饰模式的decorator的运用，给对应的
method增加新的功能；bean的事件定义，事件监听和发送等。 
JAVAEE7中，CDI规范升级到了Contexts and Dependency Injection（JSR-346）。 
5.1 CDI新特性 
相对于CDI1.0规范，CDI 1.1在一些功能进行了改进，改进点如下： 
1. 加入全局拦截器、全局装饰器和使用@Priority 注解实现替代选择。 
2. 给注解为@AroundConstruct 的构造器加入生命周期回调支持。 
3. 允许绑定拦截构造器。 
4. 拦截器绑定移入拦截器规范，以供其它规范使用。 
5. Beans 内建支持装饰器加入EventMetadata 允许事件元数据检查。 
6. 加入@Vetoed 注解让Classes 失去原有程序能力。 
7. 强化beans功能的改进，包括 @TransientReference注解允许保留实力给调用它的方法或构造函数 。 
8. 加入激活和销毁周期的事件回调 。 
9. 加入AlterableContext 允许bean 实例被明确的销毁。 
10. 给bean 在周期注解、EJB、在JavaEE 中自启动CDI 功能。 
11. 在 bean.xml 中加入class 过滤排除，以防止扫描classes 和package。 
12. 加入Unmanaged 允许方便的访问无上下文的bean 实例。 
13. 加入AfterTypeDiscovery 事件，允许在类型发现之后扩展附加注册。 
14. 加入@WithAnnotation 注解作为一种改善负载性能的方式。

Second.class})

    private String password;

如果First分组验证通过了，接着去验证Second分组，最后指定User.class表示那些没有分组的在最后。这
样就可以实现按顺序验证分组了。

第12章 CGI使用 
CGI(Common Gateway Interface)为web服务器定义了一种与外部内容生成程序交互的方法，这些程序
通常被称为CGI程序或CGI脚本。 
CGI支持是使用servlet类com.tongweb.catalina.servlets.CGIServlet实现的，通常这个servlet被映
射到"/cgi-bin/*"。在TongWeb中默认禁用CGI支持，当使用TongWeb作为HTTP服务器并需要CGI支持时，可
以添加CGI支持。 
12.1 启用CGI 
在默认的${TongWeb_HOME}/conf/default-web.xml文件中有注释掉的示例CGI servlet。要在web应用
程序中启用CGI支持，可将示例CGI servlet复制到web应用程序的WEB-INF/web.xml文件中。部署后的应用
在${TongWeb_HOME}/deployment/目录下。 
取消${TongWeb_HOME}/conf/default-web.xml文件中的CGI servlet示例的注释，可以同时为所有已部
署的web应用程序启用CGI。 
注意： 
CGI脚本用于执行TongWebJVM外部的程序。如果正在使用Java SecurityManager，则将绕过在
tongweb.policy中的安全策略配置。
