# LiuAgent 未来完善与优化路线图 (Roadmap)

基于目前 LiuAgent 已经具备的坚实基础（StateGraph 架构、混合 RAG、E2B 沙箱、Mem0 长期记忆、Langfuse 观测），为了将其打造成一个**工业级、可商业化**的超级 Agent，并在你的简历中作为杀手锏项目，建议按照以下四个维度进行深度优化。

---

## 🚀 阶段一：工程化与鲁棒性提升 (Engineering & Robustness)

目前 Agent 的“快乐路径 (Happy Path)”已经跑通，但面对真实用户的各种异常操作，还需要增强健壮性。

### 1. 结构化输出与反思机制升级
*   **现状**：目前 `reflection_node` 的验证逻辑较弱，强依赖 LLM 的自然语言输出。
*   **优化**：
    *   引入 **Pydantic** 强制约束大模型的输出格式（Structured Output）。
    *   **实现真正的 Self-Correction**：在工具调用失败时，将具体的 Exception Traceback 连同上一次的代码一并喂给 LLM，强制要求其在 JSON 中输出 `{"bug_cause": "...", "fixed_code": "..."}`。
    *   **死循环保护**：在 `AgentState` 中增加 `retry_count` 字段，当某个节点重试超过 3 次时，强制中断并降级为“请求人类介入 (Human-in-the-loop)”。

### 2. 流式交互的极致体验 (Streaming UI)
*   **现状**：Streamlit 页面使用折叠框展示工具状态，但缺乏打字机般的连贯感。
*   **优化**：
    *   将后端的 SSE 升级为真正的 **WebSocket**。
    *   在前端实现类似 Cursor/ChatGPT 的**“思考中”动画**（Think 标签页），实时将 LLM 的中间推理过程 (Thought Process) 以流式字符打印出来，消除用户的等待焦虑。

---

## 🧠 阶段二：RAG 的深度进化 (Advanced RAG)

中间件问答是本项目的核心业务场景，RAG 的准确率直接决定了产品的生死。

### 1. 引入 Reranker 模型 (重排层)
*   **痛点**：目前的 RRF (Reciprocal Rank Fusion) 只是简单的算法融合，无法理解段落与问题的深层语义匹配度。
*   **优化**：在双路召回（Vector + BM25）之后，接入一个轻量级的 **Cross-Encoder Reranker**（如 `bge-reranker-v2-m3`）。先召回 20 个片段，再通过 Reranker 精确打分，只保留最相关的 Top-5 给大模型。

### 2. 知识库的结构化切分 (Chunking Strategy)
*   **痛点**：如果 PDF 是按固定字数（如 500 字）切分的，很容易把一段完整的配置代码或表格切断。
*   **优化**：
    *   实现 **Semantic Chunking（语义切分）** 或 **Markdown/Title-based Chunking**。确保每个 Chunk 都是一个完整的逻辑单元（例如：一个配置项的所有说明都在一个 Chunk 里）。
    *   引入 **Parent-Child Retriever**：检索时匹配子段落，但喂给大模型时提供完整的父章节上下文。

## 🛠️ 阶段三：多模态与工具链扩展 (Multimodal & Toolchain)

让 Agent 变成真正的“全栈工程师”和“运维专家”。

### 1. 运维自动化闭环
*   **现状**：目前 Agent 只能看文档和读日志。
*   **优化**：增加**“只读权限的系统执行工具”**。例如，允许 Agent 通过 SSH 连接到测试服务器，执行 `top`、`df -h`、`netstat` 或抓取 TongWeb 的运行时 JMX 指标，实现“诊断 -> 查文档 -> 给出修复脚本”的全自动运维闭环。

### 2. 复杂数据可视化看板
*   **现状**：目前只能生成单张静态图片保存在本地。
*   **优化**：
    *   利用 E2B 沙箱生成动态的 HTML/JS 图表（如 ECharts 或 Plotly）。
    *   在 Streamlit 前端直接渲染这些交互式图表，让用户可以缩放、悬停查看数据细节。

---

## 📊 阶段四：数据飞轮与持续评估 (Data Flywheel & Eval)

为了在面试中展示你具备“AI 产品经理”的思维，你需要建立一套评估体系。

### 1. 构建黄金测试集 (Golden Dataset)
*   从日常提问中筛选出 50 个典型的中间件疑难问题（含报错日志、配置诉求等），并人工编写标准答案（Ground Truth）。

### 2. LLM-as-a-Judge 自动化评估
*   **实施**：编写一个评估脚本，定期运行这 50 个问题。使用一个强大的模型（如 GPT-4o 或 Qwen-Max）来对比 Agent 的回答与标准答案，从“准确性”、“幻觉率”、“相关性”三个维度打分。
*   **价值**：每次修改 RAG 逻辑或 Prompt 后，跑一遍自动化测试，用数据证明你的优化是有效的（例如：准确率从 75% 提升至 92%）。

### 3. 基于 Langfuse 的用户反馈闭环
*   在前端界面每条回答下方添加 `👍 / 👎` 按钮。
*   将用户的点踩记录（包含当时的 Trace、检索到的文档）汇总到一个看板中。定期 Review 这些 Bad Case，如果是文档没写，就补充文档；如果是没搜到，就优化检索参数。
