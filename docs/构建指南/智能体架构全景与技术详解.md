# 智能体 (AI Agent) 架构全景与技术详解

一个成熟的智能体系统通常被划分为**核心智能**（传统定义）和**工程系统**（生产落地）两个层面。以下是各部分的详细功能及实现技术栈：

## 第一部分：核心智能 (The Cognitive Core)
> **关注点**：智能体的推理与交互能力（解决“能不能做”的问题）。

1.  **大脑 (Brain) - 大语言模型 (LLM)**
    *   **角色**: 核心控制器，负责处理信息、意图理解、逻辑推理、决策制定。
    *   **关键技术详解**:
        *   **基础模型 (Foundation Models)**:
            *   *Closed Source*: **GPT-4 / Claude 3.5 / Gemini 1.5**: 闭源 SOTA (State of the Art) 模型，逻辑最强，适合充当 Agent 的“主脑”。
            *   *Open Source*: **Llama 3 / Qwen 2.5 / DeepSeek-V3**: 开源界的最强模型，支持私有化部署，适合对数据隐私要求高的场景。
        *   **推理框架 (Inference)**:
            *   **开源推理库**: **vLLM**: 最流行的开源推理库，利用 PagedAttention 技术极大提升了吞吐量，是部署开源模型的首选。
            *   **本地运行工具**: **Ollama**: 本地运行大模型的工具，封装极好，适合开发者在笔记本上快速测试 Llama3 等模型。
            *   **高性能推理库**: **TensorRT-LLM**: NVIDIA 推出的高性能推理库，针对自家 GPU 做了极致优化，生产环境追求极致低延迟时的选择。
        *   **微调 (Fine-tuning)**:
            *   **参数高效微调**: **LoRA (Low-Rank Adaptation)**: 一种高效微调技术，只训练极少量的参数（旁路矩阵），大大降低了微调大模型的显存门槛。
            *   **量化微调**: **QLoRA**: 在 LoRA 基础上引入 4-bit 量化，进一步降低显存需求，单张消费级显卡即可微调大参数模型。

2.  **规划 (Planning)**
    *   **角色**: 任务拆解与反思。
    *   **关键技术详解**:
        *   **提示工程 (Prompt Engineering)**:
            *   **思维链**: **CoT (Chain of Thought)**: 思维链。要求模型在给出答案前先写出推理步骤，显著提升逻辑/数学能力。
            *   **思维树**: **ToT (Tree of Thoughts)**: 思维树。让模型探索多种可能的推理路径（分支），并对每个分支进行自我评估，选择最佳路径或回溯。比线性 CoT 更强大，适合需要探索和试错的复杂问题（如创意写作、复杂规划）。
            *   **推理与行动**: **ReAct**: Reason + Act。模型交替进行“思考-行动-观察”，是 Agent 调用工具的标准范式。
        *   **编排框架 (Orchestration)**:
            *   **通用开发框架**: **LangChain**: 最早也是最全的 Agent 开发框架，提供了从 Prompt 模板到记忆管理的各种组件。
            *   **图编排框架**: **LangGraph**: LangChain 的进阶版，基于“图论”设计。它允许开发者定义循环（Loop）和条件分支，非常适合构建需要长时间运行、能够自我纠错的复杂 Agent。
            *   **多智能体框架**: **AutoGen**: 微软推出的多 Agent 框架。它擅长让多个角色不同的 Agent（如“程序员”和“产品经理”）互相对话、协作来解决问题。
            *   **应用集成SDK**: **Semantic Kernel**: 微软推出的 SDK，旨在将 LLM 能力轻松集成到现有的 C#/Python 代码项目中，强调“Kernel”作为核心协调者，连接“Plugin”（工具）和“Planner”（规划器）。

    *   **👉 选型指南 (How to Choose)**:
        *   **LangChain**: 适合**简单、线性**的任务。如果你只是想做一个“PDF 问答助手”或“简单的 API 调用器”，用 LangChain 的 `LCEL` (LangChain Expression Language) 就够了，开发快，上手简单。
        *   **LangGraph**: 适合**复杂、有状态、需要纠错**的生产级 Agent。如果你需要 Agent 在出错时重试、在某些条件下跳转步骤、或者需要持久化保存 Agent 的状态（Human-in-the-loop），**必须**使用 LangGraph。它是构建真正“智能”Agent 的基石。
        *   **AutoGen**: 适合**多角色协作 (Multi-Agent)** 的场景。如果你需要模拟一个软件开发团队（一个 Agent 写代码，一个 Agent 跑测试，一个 Agent 提修改意见），AutoGen 的“对话流”模式是目前最自然的实现方式。

3.  **记忆 (Memory)**
    *   **角色**: 存储历史交互和外部知识。
    *   **关键技术详解**:
        *   **向量数据库 (Vector DB) 选型指南**:
            *   **Milvus / Zilliz**: **大规模生产首选**。云原生架构，支持十亿级向量，功能最全。
            *   **Qdrant**: **高性能首选**。Rust 编写，极快，支持复杂的 Metadata 过滤（如“只搜 2024 年的文档”），开发体验极佳。
            *   **Elasticsearch (8.0+)**: **混合检索首选**。如果公司已有 ES 集群，直接用它。原生支持 BM25 + Vector，省去维护两套系统的麻烦。
            *   **Chroma**: **轻量级/开发首选**。Python 原生，安装极简，适合 Notebook 实验和小型应用。
            *   **pgvector**: **Postgres 用户首选**。如果业务数据在 PG 里，直接装个插件就能存向量，事务一致性最好。
        *   **图数据库 (Graph DB)**:
            *   **业界标准图库**: **Neo4j**: 业界最成熟的图数据库。在 RAG 中用于构建 **GraphRAG**，不仅检索文档片段，还能检索实体间的关系（如“A 是 B 的子公司”），解决跨文档推理问题。
        *   **嵌入模型 (Embedding) 选型指南**:
            *   **通用商业模型**: **OpenAI text-embedding-3-small/large**: 行业基准。**优点**: 稳定、多语言支持好。**缺点**: 数据出境、按 Token 收费。
            *   **开源最强模型**: **BGE-M3 (BAAI)**: 目前最强的开源 Embedding。**优点**: 支持多语言、支持长文本 (8k)、同时输出稠密/稀疏/多向量 (ColBERT)，一鱼多吃。
            *   **长文本模型**: **Jina-Embeddings-v3**: 支持 8k 上下文，专门针对长文档优化，支持通过 LoRA 适配特定领域。
            *   **代码专用模型**: **Voyage-Code-2**: 专门针对代码检索优化，理解函数定义和调用关系，做“代码问答”必选。
            *   **多语言模型**: **Cohere Embed v3**: 商业模型，但在非英语（尤其是欧洲语言）表现优于 OpenAI。

4.  **工具使用 (Tool Use / Action)**
    *   **角色**: 连接外部世界。
    *   **关键技术详解**:
        *   **通用连接协议**: **MCP (Model Context Protocol)**: Anthropic 推出的开放协议，旨在标准化 LLM 与数据/工具的连接。只需按标准写一个 Server，所有支持 MCP 的客户端（Claude Desktop, Cursor 等）都能通用，解决了“每个应用都要重写集成代码”的痛点。
        *   **结构化调用能力**: **Function Calling**: OpenAI 引领的 API 能力，让 LLM 输出结构化的 JSON 来调用函数，而不是模糊的自然语言。
        *   **接口描述标准**: **OpenAPI (Swagger)**: 业界标准的 REST API 描述格式。将 OpenAPI 文档喂给 LLM，它就能自动理解如何调用这个 API（路径、参数、方法），是实现“零代码对接工具”的基础。
        *   **沙箱 (Sandbox)**:
            *   **云端代码沙箱**: **E2B**: 专为 AI 代码执行设计的云端沙箱，安全且启动极快。
            *   **容器化隔离**: **Docker**: 容器化技术，常用于在本地构建隔离环境，防止 Agent 生成的恶意代码破坏宿主机。

## 第二部分：工程系统 (The Engineering System)
> **关注点**：稳定性、安全、成本与运维（解决“好不好用”和“敢不敢用”的问题）。

5.  **可观测性 (Observability)**
    *   **角色**: 调试与监控。
    *   **关键技术详解**:
        *   **官方监控平台**: **LangSmith**: LangChain 官方的监控平台，提供可视化的 Trace（调用链）查看，是调试复杂 Agent 逻辑的神器。
        *   **开源观测平台**: **Arize Phoenix**: 开源的 LLM 可观测性平台，特别擅长对 Trace 数据进行 Embedding 可视化分析，帮助发现潜在的聚类问题。
        *   **行业追踪标准**: **OpenTelemetry (OTel)**: 分布式追踪的行业标准协议，现在被用于统一 LLM 应用的监控数据格式。
        *   **评估 (Evaluation)**:
            *   **RAG评估框架**: **Ragas**: 专门用于评估 RAG 检索质量（相关性、忠实度）的框架。
            *   **单元测试框架**: **DeepEval**: 提供了一套类似单元测试（Unit Test）的 API，可以在 CI/CD 流程中自动测试 LLM 的输出质量。
            *   **Prompt测试工具**: **Promptfoo**: 一个 CLI 工具，用于批量测试 Prompt 的效果，支持对比不同模型在同一 Prompt 下的表现。

6.  **安全与护栏 (Security & Guardrails)**
    *   **角色**: 防御与合规。
    *   **关键技术详解**:
        *   **结构验证库**: **Guardrails AI**: 一个 Python 库，通过定义 Pydantic 数据模型来强制验证 LLM 的输出结构和内容，验证失败会自动触发重试。
        *   **对话护栏系统**: **NVIDIA NeMo Guardrails**: 专门针对对话系统的护栏，使用 Colang 语言编写规则，可以严格控制对话流程（例如：用户问 A，强制引导回 B）。
        *   **注入防御API**: **Lakera Guard**: 专注于防御 Prompt Injection（提示词注入）的 API 服务，维护了最新的攻击样本库。
        *   **隐私保护工具**: **Microsoft Presidio**: 隐私保护工具，利用 NLP 技术自动识别并擦除文本中的 PII（个人身份信息）。

7.  **知识库工程 (Knowledge Engineering / RAG)**
    *   **角色**: 提供高质量私有数据。
    *   **关键技术详解**:
        *   **ETL 与切片 (Chunking) 策略**:
            *   **固定大小切片 (Fixed-size)**: **最简单**。每 500 个字切一段。**缺点**: 容易把一个完整的句子或逻辑切断。
            *   **递归字符切片 (Recursive Character)**: **LangChain 默认**。先按段落 (`\n\n`) 切，太长了再按句子 (`.`) 切。**推荐**作为默认起点。
            *   **语义切片 (Semantic Chunking)**: **进阶**。计算前后两句的 Embedding 相似度，如果突变（如从讲“配置”跳到讲“报错”），就在这里切开。保证每块内容语义纯净。
            *   **父子索引 (Parent-Child)**: 索引时切成小块（Small Chunk）以便精准匹配，但召回时返回该小块所属的大块（Parent Chunk）给 LLM。**推荐**用于复杂技术文档。
            *   **Agentic Chunking**: **最贵但最好**。让 LLM 读完全文，自己决定怎么拆分章节。适合处理格式极其不规范的文档。
            *   **通用文档解析**: **Unstructured.io**: 强大的文档解析库，支持几乎所有格式。
            *   **复杂表格解析**: **LlamaParse**: LlamaIndex 推出的解析器，专门优化了 PDF 中的**复杂表格**识别，这是传统解析器的弱项。
        *   **检索策略**:
            *   **基础检索算法**:
                *   **关键词检索**: **BM25 / TF-IDF**: 基于词频的传统算法。**优势**: 对专有名词（错误码、产品型号）匹配极准。**劣势**: 无法理解语义（"苹果" vs "水果"）。
                *   **稠密向量检索 (Dense Retrieval)**: **Embedding**: 将文本转为向量。**优势**: 理解语义。**劣势**: 对精确匹配（如 `TW-10042`）效果不佳。
                *   **稀疏向量检索 (Sparse Retrieval)**: **SPLADE**: 结合了 BM25 的精确性和 Embedding 的语义能力，生成稀疏的语义向量。
            *   **高级检索算法 (Advanced RAG)**:
                *   **混合检索 (Hybrid Search)**: **CC (Common Practice)**: 同时执行 BM25 和 Vector Search，然后使用 **RRF (Reciprocal Rank Fusion)** 算法融合排名。**工业界标配**。
                *   **假设文档嵌入 (HyDE)**: **Hypothetical Document Embeddings**: 先让 LLM 生成一个“虚构的完美答案”，然后用这个虚构答案去检索。**适用场景**: 用户问题很短，但目标文档很长，解决语义空间不对齐问题。
                *   **父子索引 (Parent-Child Indexing)**: **Small-to-Big**: 索引时切分小块（Sentence/Chunk），检索时返回该小块所属的大块（Parent Document）。**优势**: 检索精准（小块语义集中），但生成时上下文完整（大块信息丰富）。
                *   **句子窗口检索 (Sentence Window Retrieval)**: 检索到特定句子后，自动扩展返回其前后 N 个句子。**优势**: 保持上下文连贯性。
                *   **多路查询 (Multi-Query / Query Expansion)**: 利用 LLM 将用户的一个问题改写成 3-5 个不同角度的问题，并行检索，取并集。**优势**: 提高召回率 (Recall)。
        *   **重排序 (Rerank)**: **高精度重排模型**: **Cohere Rerank / BGE-Reranker**。在粗排检索出 50 条结果后，用一个精排模型（Cross-Encoder，计算量大但极准）逐一打分，选出最相关的 5 条。这是提升 RAG 准确率性价比最高的手段。

8.  **模型运维 (LLMOps)**
    *   **角色**: 降本增效。
    *   **关键技术详解**:
        *   **网关/路由**:
            *   **统一API网关**: **LiteLLM**: Python 库，统一了 100+ 模型厂商的 API 调用格式。
            *   **开源聚合网关**: **OneAPI**: 一个开源的聚合 API 网关，支持渠道管理、负载均衡，适合企业内部统一管理所有 LLM Key。
        *   **缓存**:
            *   **语义缓存层**: **GPTCache / Redis Semantic Cache**: 语义缓存层。如果用户问了重复或相似的问题，直接返回缓存答案，节省昂贵的 Token 费。
        *   **部署**:
            *   **Python服务框架**: **Ray Serve**: 适合 Python 生态的模型服务框架，支持动态扩缩容，适合部署复杂的 LLM 组合应用。
            *   **容器编排标准**: **Kubernetes (K8s)**: 容器编排标准，用于在大规模集群中管理 LLM 服务的生命周期。

## 第三部分：工业级 Agent 处理全流程 (The Industrial Agent Workflow)
> **关注点**：从用户输入到最终响应的完整生命周期，涵盖每一步的核心算法与处理逻辑。

一个生产环境的 RAG/Agent 请求通常会经历以下 8 个阶段：

### Phase 1: 接入与护栏 (Ingestion & Guardrails)
**目标**: 拒绝恶意请求，保护系统安全。
1.  **输入审计**: 使用 **NVIDIA NeMo Guardrails** 或 **Presidio** 扫描用户输入。
    *   *PII 检测*: 是否包含手机号、身份证？ -> 掩码处理。
    *   *Jailbreak 检测*: 是否包含“忽略所有指令”等越狱 Prompt？ -> 直接拒绝。
2.  **意图初筛**: 快速判断是否为闲聊 (Chit-chat)。
    *   *算法*: 简单的 Embedding 相似度匹配或轻量级分类模型 (BERT)。
    *   *动作*: 如果是“你好”，直接回复，不消耗昂贵的 RAG 资源。

### Phase 2: 查询理解与增强 (Query Understanding & Expansion)
**目标**: 弥合“用户怎么说”和“文档怎么写”之间的鸿沟。
1.  **查询改写 (Query Rewriting)**:
    *   *问题*: 用户说“它启动报错了”。系统不知道“它”是谁。
    *   *算法*: 利用历史对话 Session，让 LLM 重写为独立完整句：“TongWeb Server 在 Linux 环境下启动报错”。
2.  **查询扩展 (Query Expansion)**:
    *   *算法*: **Multi-Query**。LLM 生成 3 个变体：“TongWeb 启动日志分析”、“TongWeb 常见启动错误”、“TongWeb 启动失败解决方案”。
    *   *算法*: **HyDE**。LLM 生成一个假设的答案段落，用该段落的向量去检索。

### Phase 3: 路由分发 (Routing)
**目标**: 选择最合适的解决路径。
1.  **语义路由 (Semantic Router)**:
    *   *算法*: 计算用户 Query 向量与预定义 Route 向量（如“技术支持”、“售前咨询”、“代码编写”）的 Cosine 相似度。
    *   *决策*: 
        *   -> 匹配到“数据库问题” -> 路由到 SQL Agent。
        *   -> 匹配到“文档问题” -> 路由到 RAG Pipeline。
        *   -> 匹配到“复杂任务” -> 路由到 LangGraph Agent（多步规划）。

### Phase 4: 混合检索 (Hybrid Retrieval)
**目标**: 高召回率 (Recall)，确保相关文档都在候选集中。
1.  **并行执行**:
    *   **路 A (Sparse)**: **BM25** 检索。输入：“TW-10042”。输出：文档 A (包含精确错误码)。
    *   **路 B (Dense)**: **Embedding (BGE-M3)** 检索。输入：向量化后的 Query。输出：文档 B (语义相关)。
2.  **结果融合**:
    *   **算法**: **RRF (Reciprocal Rank Fusion)**。公式：`score = 1 / (k + rank_BM25) + 1 / (k + rank_Vector)`。
    *   *效果*: 既照顾了精确匹配，又照顾了语义匹配，生成 Top-50 候选集。

### Phase 5: 重排序与过滤 (Rerank & Filtering)
**目标**: 高准确率 (Precision)，剔除噪音。
1.  **精排 (Reranking)**:
    *   *模型*: **BGE-Reranker-v2-m3** (Cross-Encoder 架构)。
    *   *动作*: 将 User Query 与 Top-50 文档逐一对拼，输入模型打分。
    *   *输出*: 重新排序后的 Top-5 文档。
2.  **阈值截断 (Thresholding)**:
    *   *规则*: 如果 Top-1 的分数 < 0.3，说明知识库里根本没有相关文档。
    *   *动作*: 触发“未找到相关信息”的兜底回复，避免强行回答导致幻觉。

### Phase 6: 上下文构建 (Context Construction)
**目标**: 组装给 LLM 的最终 Prompt。
1.  **上下文压缩**:
    *   *算法*: **LongLLMLingua**。如果检索到的文档太长，使用小模型识别并删除无关的句子，只保留高与 Query 相关的片段，节省 Token。
2.  **Prompt 组装**:
    *   *模板*: 
        ```
        基于以下已知信息（Context）：
        [文档1]...
        [文档2]...
        请回答用户问题（Question）：...
        如果不确定，请说“我不知道”。
        ```

### Phase 7: 生成与流式输出 (Generation & Streaming)
**目标**: 生成最终答案。
1.  **调用 LLM**: DeepSeek-V3 / GPT-4。
2.  **流式响应 (Streaming)**: 收到一个 Token 就推给前端一个，降低用户的心理等待时间 (TTFT)。

### Phase 8: 幻觉检测与反馈 (Validation & Feedback)
**目标**: 确保回答真实可信。
1.  **自查 (Self-Correction)**:
    *   *算法*: **Self-RAG**。让 LLM 在生成答案的同时输出 `<citation>` 标签。如果生成的引用在文档中找不到，强制重写。
2.  **后处理**:
    *   *格式修复*: 如果要求输出 JSON 但 LLM 输出了 Markdown，使用解析器自动修复。
