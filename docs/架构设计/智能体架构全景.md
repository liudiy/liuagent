# 智能体 (AI Agent) 架构全景
---

## 1. 大脑 (Brain) - 认知架构与编排
如果说 LLM 是 CPU，那么编排框架就是操作系统。它决定了 Agent 如何思考、规划以及执行任务。

### 1.1 LangGraph (基于图论的状态机)
**核心理念**：将 Agent 的运行过程建模为一个**有状态的图 (Stateful Graph)**。

#### 深度解剖：它是如何工作的？
传统的 LLM 调用是线性的（Input -> Output），但真实的 Agent 需要循环（思考 -> 行动 -> 观察 -> 再思考）。LangGraph 通过以下三个核心概念实现了这一点：

1.  **State (状态)**：这是 Agent 的“短期记忆”。
    *   在代码层面，它通常是一个 Python 的 `TypedDict`。
    *   **Append 机制**：不同于普通的变量赋值，LangGraph 的状态更新通常采用“追加”模式。当一个节点（Node）产生新消息时，它不会覆盖旧消息，而是通过 `operator.add` 将其追加到历史记录中。这确保了 Agent 在多轮循环中不会忘记之前的步骤。

2.  **Node (节点)**：这是 Agent 的“动作单元”。
    *   一个节点就是一个 Python 函数，比如“调用 LLM”、“执行 Python 代码”或“搜索网络”。每个节点接收当前的 State，执行操作，然后返回新的数据来更新 State。

3.  **Edge (边) 与 Conditional Edge (条件边)**：这是 Agent 的“决策逻辑”。
    *   **路由机制**：条件边本质上是一个路由函数（Router）。它查看 LLM 的输出，决定下一步去哪里。
    *   *示例*：如果 LLM 输出包含 `<tool_call>`，路由函数就会把流程指向 `ToolNode`；如果 LLM 输出是普通文本，则指向 `END` 结束运行。

4.  **Checkpointer (持久化)**：这是 Agent 的“存档点”。
    *   每执行一步，LangGraph 都会把当前的 State 序列化并存入数据库（如 Postgres）。
    *   **价值**：这使得我们可以实现“时间旅行”。比如 Agent 跑错了，你可以暂停它，修改数据库里的状态（纠正它的错误记忆），然后让它从断点处继续运行。

#### 核心操作与机制深度解析 (Mechanisms Deep Dive)
LangGraph 的核心在于**控制流的编排**，它将代码逻辑具象化为一张可遍历的图。

##### 1. Define & Compile (图的构建与编译)
这是 Agent 生命周期的“设计阶段”。

*   **Key Terms (关键术语)**
    *   **StateGraph**: 整个图的容器，它规定了图的“血脉”（即 `State` Schema）。
    *   **Nodes**: 静态的处理单元，通常是纯函数（Input State -> Output State Update）。
    *   **Edges**: 动态的流向规则。
    *   **Checkpointer**: 负责快照存储的中间件（如 `MemorySaver`）。

*   **How it works (工作原理 - 像素级流程还原)**
    `compile()` 是从代码到可执行对象的关键转换：
    1.  **Validation (静态检查)**: 扫描所有注册的 Node 和 Edge。
        *   *Check 1*: 是否有孤岛节点（Unreachable Nodes）？
        *   *Check 2*: 是否有死胡同（Nodes with no outgoing edges）？
    2.  **Topology Sort (拓扑构建)**: 将节点和边解析为邻接矩阵（Adjacency Matrix）。
    3.  **Runnable Creation (运行时生成)**: 实例化 `CompiledGraph` 对象。
        *   这个对象继承自 LangChain 的 `Runnable` 协议。
        *   它内置了一个微型状态机引擎，准备好接收 `invoke/stream` 调用。

##### 2. Invoke / Stream (图的运行循环)
这是 Agent 的“运行阶段”。

*   **Key Terms (关键术语)**
    *   **Pregel Loop**: LangGraph 底层采用的类 Pregel 算法循环。
    *   **Thread ID**: 类似于操作系统的进程 ID，用于区分不同的对话线程。
    *   **StreamMode**: 输出模式（`values` 吐出完整状态，`updates` 吐出增量变更）。

*   **How it works (工作原理 - 像素级流程还原)**
    LangGraph 的运行其实是一个无限的 **Pregel 循环**：
    1.  **Initialize (启动)**: 用户调用 `invoke(input)`，Input 被推入任务队列。
    2.  **Load State (读档)**: 
        *   系统根据 `thread_id` 查询 Checkpointer。
        *   如果存在历史记录，恢复到上次的 `State`；如果是新对话，初始化为空 State。
    3.  **Task Fetching (取任务)**: 从队列中取出当前需要执行的 Node（例如 `StartNode` 或 `ToolNode`）。
    4.  **Execution (执行)**: 
        *   运行 Node 对应的 Python 函数。
        *   输入是当前 `State`。
        *   输出是 `Update Dict`（增量更新）。
    5.  **Reduction (归约)**: 
        *   **CRITICAL**: 系统**不会**直接覆盖 State。
        *   它调用预定义的 `Reducer` 函数（如 `operator.add`），将 `Update Dict` 与 `Current State` 合并。
        *   *示例*: `OldMessages + NewMessage = AllMessages`。
    6.  **Persistence (存档)**: 合并后的新 State 被序列化，写入数据库（Checkpointer）。
    7.  **Routing (路由)**: 
        *   系统查看 Conditional Edges。
        *   如果是 `END`，循环结束。
        *   如果是其他 Node，将该 Node 推入任务队列，跳转回步骤 3。

##### 3. Time Travel (状态干预)
LangGraph 最独特的能力，允许你“修改历史”。

*   **How it works (工作原理)**
    1.  **Get**: `app.get_state(config)` 直接读取数据库中该 `thread_id` 的最新快照。
    2.  **Update**: `app.update_state` 不仅仅是改变量，它实际上是在当前的执行历史中**插入了一个新版本**（Fork）。系统会认为“这一步是人工修正过的”，并基于这个新状态继续运行。

### 1.2 AutoGen (多智能体对话)
**核心理念**：**世界是一个巨大的聊天室**。任何问题都可以通过多个智能体之间的“对话”来解决。

#### 深度解剖：它是如何工作的？
AutoGen 的设计灵感来源于人类团队的协作模式。它不强调复杂的图结构，而是强调**角色扮演**和**消息传递**。

1.  **ConversableAgent (可对话智能体)**：
    *   这是所有 Agent 的基类。每个 Agent 都有 `send()` 和 `receive()` 方法，就像我们在微信上发消息一样。
    *   **Reply Hook**：当 Agent 收到消息时，它不会立即回答，而是触发一系列注册好的“回复函数”。比如：先检查“我需要运行代码吗？”，再检查“我需要调用 LLM 吗？”。

2.  **UserProxy (用户代理) vs Assistant (助手)**：
    *   **Assistant**：负责思考和写代码。它通常连接着强大的 LLM（如 GPT-4）。
    *   **UserProxy**：负责**执行代码**。它像一个沙箱环境（Sandbox）。
    *   **自动反馈循环**：这是一个非常精彩的设计。Assistant 写出 Python 代码 -> UserProxy 捕获代码并在本地（或 Docker）运行 -> 运行结果（Stdout/Stderr）被当做一条新消息发回给 Assistant。
    *   *结果*：Assistant 看到报错信息，会自己修改代码，再次发送给 UserProxy，直到代码运行成功。这就是“自我修复”的原理。

#### 核心操作与机制深度解析 (Mechanisms Deep Dive)
AutoGen 的本质是**基于事件驱动的异步通信系统**。

##### 1. Initiate Chat (对话启动)
这是点燃引擎的操作。

*   **Key Terms (关键术语)**
    *   **Sender / Receiver**: 发起方与接收方。
    *   **Context**: 初始上下文（System Prompt 或首条消息）。
    *   **Max Turns**: 安全护栏，防止无限循环对话消耗 Token。

*   **How it works (工作原理 - 像素级流程还原)**
    1.  **Preparation**: UserProxy 准备好 `message` (Input)。
    2.  **Send**: 调用 `send(message, recipient=Assistant)`。
    3.  **Append**: 消息被追加到 UserProxy 的 `outbox` 和 Assistant 的 `inbox` (即 `chat_messages` 列表)。
    4.  **Trigger**: Assistant 的 `receive()` 方法被触发，自动进入 **Reply Loop**。

##### 2. Reply Registration (Hook 机制)
这是 AutoGen 灵活性的来源。它允许你劫持 Agent 的大脑。

*   **Key Terms (关键术语)**
    *   **Trigger**: 触发条件（例如：只针对来自 `UserProxy` 的消息触发）。
    *   **Reply Function**: 实际的处理逻辑。
    *   **Position**: 优先级。你可以注册多个回复函数，它们像洋葱一样包裹着核心 LLM。

*   **How it works (工作原理 - 像素级流程还原)**
    当 Agent 需要回复时，它**不是**直接调用 LLM，而是遍历一个**责任链 (Chain of Responsibility)**：
    1.  **Hook 1 (Termination Check)**: 
        *   检查收到的消息是否包含 "TERMINATE"。
        *   如果是 -> 停止对话，返回 None。
    2.  **Hook 2 (Function Call Execution)**:
        *   检查收到的消息是否包含代码块或工具调用请求。
        *   如果是 -> 执行代码/工具 -> 返回执行结果 (String) -> **中断链条** (不再问 LLM，直接把结果发回去)。
    3.  **Hook 3 (Human Input)**:
        *   (如果开启了 `human_input_mode`) 暂停程序，等待你在控制台输入。
        *   有输入 -> 返回你的输入 -> **中断链条**。
    4.  **Hook 4 (LLM Generation)**:
        *   (这是最后的兜底) 构造 Prompt -> 调用 OpenAI API -> 返回 LLM 的回答。
    *   **机制**: 这种设计确保了**行动优先于说话**。如果能执行代码，就先执行代码，而不是废话。

##### 3. Code Execution (代码执行沙箱)
AutoGen 的杀手锏。

*   **How it works (工作原理 - 像素级流程还原)**
    1.  **Block Extraction (正则提取)**: `UserProxy` 收到消息后，使用正则表达式扫描文本。
        *   Target: ````python ... ```` 包裹的代码块。
    2.  **File Writing (落盘)**: 
        *   系统将提取出的代码写入一个临时文件（例如 `tmp_code_3a1b.py`）。
        *   *Why?* 为了模拟真实的 Python 文件执行环境。
    3.  **Process Forking (子进程执行)**:
        *   系统调用 `subprocess.run(["python", "tmp_code_3a1b.py"])`。
        *   或者（如果配置了 Docker），`docker exec container_id python tmp_code.py`。
    4.  **Capture (捕获)**:
        *   同时捕获 `stdout` (正常输出) 和 `stderr` (报错信息)。
    5.  **Return (返回)**: 将捕获的文本封装为一条新的 Message，发送回给 Assistant。

## 2. 记忆 (Memory) - 跨越时间的连贯性
LLM 本身是无状态的（Stateless），它记不住你上一秒说了什么。记忆模块就是 Agent 的海马体。

### 2.1 Mem0 (混合记忆架构)
**核心理念**：基于 Google 白皮书与官方文档定义的**四层记忆体系**，实现从瞬时对话到组织级知识的全覆盖。

#### 深度解剖：它是如何工作的？
Mem0 通过**混合检索（Vector + Graph）**的底层技术，支撑了以下四个维度的记忆层级，解决了传统 RAG 缺乏上下文连贯性的痛点：

1.  **Conversation Memory (对话记忆)**：
    *   **定义**：In-flight messages inside a single turn. (单轮交互中的即时消息)。
    *   **作用**：暂存当前正在进行的对话片段（如你刚发出的指令），用于维持最基本的上下文连贯性。这通常存储在高速缓存（如 Redis）中。

2.  **Session Memory (会话记忆)**：
    *   **定义**：Short-lived facts that apply for the current task or channel. (当前任务或频道内的短期事实)。
    *   **作用**：在一次完整的交互窗口（Session）内有效。例如，你在当前窗口说“我正在写一个 Python 脚本”，Mem0 会记住这个背景，直到你会话结束或主动清除。

3.  **User Memory (用户记忆)**：
    *   **定义**：Long-lived knowledge tied to a person, account, or workspace. (与特定人、账号或工作区绑定的长期知识)。
    *   **作用**：这是 Mem0 最强大的地方。它会跨 Session 记住你的偏好。比如你上个月说“我只用 PyTorch”，下个月你再来问“写个神经网络”，它会自动根据 User Memory 生成 PyTorch 代码，而不需要你重复说明。

4.  **Organizational Memory (组织记忆)**：
    *   **定义**：Shared context available to multiple agents or teams. (多个 Agent 或团队共享的上下文)。
    *   **作用**：打破数据孤岛。团队 A 沉淀的知识（如“公司服务器 IP 是 192.168.x.x”），团队 B 的 Agent 也能直接读取使用。

#### 底层技术支撑 (Graph + Vector)
Mem0 是如何实现上述记忆的**动态更新**与**关联**的？
*   **Vector Store (向量存储)**：负责**Semantic Memory (语义记忆)**。用于模糊匹配，比如通过 Embedding 找到相似的历史对话。
*   **Graph Store (图存储)**：负责**结构化关联**。
    *   *动态更新机制*：这是 Mem0 的核心优势。如果你周一说“喜欢 Java”，周三说“讨厌 Java，改用 Python”。
    *   **传统 RAG**：会保留矛盾的记录。
    *   **Mem0**：会检测到冲突，并在图数据库中**更新**代表你（User Node）与语言（Language Node）之间的关系边（Edge），确保 User Memory 始终保持最新且一致。

#### 核心操作与机制深度解析 (Mechanisms Deep Dive)
要真正掌握 Mem0，不能只看 API 调用，必须理解数据在底层的流转机制。

##### 1. Add Memory (添加记忆)
这是 Mem0 最核心的流水线，对应官方架构图的完整流程。

*   **Key Terms (关键术语)**
    *   **Messages**: 输入的原始对话。
    *   **Extraction LLM**: 负责从对话中提炼事实的第一个模型。
    *   **Update LLM**: 负责决策如何修改记忆的第二个模型。
    *   **Vector/Graph DB**: 双重存储后端。

*   **How it works (工作原理 - 像素级流程还原)**
    请想象一个分叉再汇聚的河流系统：
    1.  **Input (输入)**: 系统接收到 `Messages` (Conversations)。
    2.  **Extraction (提取)**: 消息首先送入 **Extraction LLM**。它像一个矿工，从非结构化文本中挖掘出两类黄金：
        *   *New Memories*: 适合向量检索的文本片段。
        *   *New Entities & Relations*: 适合图谱存储的节点与边。
    3.  **Dual Branching (双流分支)**:
        *   **Branch A (Vector)**: 提取出的新记忆准备存入 Vector Database。
        *   **Branch B (Graph)**: 提取出的新实体关系准备存入 Graph Database。
    4.  **Retrieval (回溯)**: **(关键步骤)** 在写入之前，系统会先从数据库中**查**。
        *   从 Vector DB 找：有没有相似的旧记忆？
        *   从 Graph DB 找：有没有已存在的实体（如 "User"）？
    5.  **Update Decision (决策)**: 将 [新提取的信息] + [查到的旧信息] 一起送入 **Update LLM**。这个模型会输出三个指令之一：
        *   **Add**: 这是全新的，存入。
        *   **Update**: 比如将“喜欢 Java”改为“喜欢 Python”。
        *   **Delete**: 删除过时的错误信息。
    6.  **Execution (落盘)**: 最终的 `Add/Update/Delete` 指令被执行，分别更新 Vector Database 和 Graph Database。

##### 2. Search Memory (搜索记忆)
检索不仅仅是查字典，而是一次复杂的混合查询。

*   **Key Terms (关键术语)**
    *   **Query**: 用户的自然语言问题。
    *   **Rewritten Query**: 经过 LLM 优化、更适合检索的查询语句。
    *   **Embedding Model**: 负责将文本转化为向量（如 OpenAI text-embedding-3）。
    *   **Dual Search**: 同时进行的向量搜索与图谱搜索。

*   **How it works (工作原理 - 像素级流程还原)**
    根据官方架构图，这是一个双路并行的漏斗模型：
    1.  **Input (输入)**: 用户输入原始查询 `Query` (例如 "我上次提到的那个 Python 库叫什么？")。
    2.  **LLM Pre-processing (预处理)**: `Query` 首先送入 LLM，LLM 承担两个任务：
        *   **Task A**: 生成 `Rewritten Query` (重写查询)，消除指代不明（将 "那个库" 补全为具体的上下文描述）。
        *   **Task B**: 提取 `Entities & Relations` (实体与关系)，用于图谱定位。
    3.  **Dual Parallel Search (双路并行搜索)**:
        *   **Path A (Vector Search)**:
            *   `Rewritten Query` -> **Embedding Model** -> `Query Vector`。
            *   在 **Vector Database** 中进行余弦相似度计算，召回 Top-K 相关的文本片段。
            *   *Result*: `[0.54, 0.89, -0.43...]` (向量匹配得分)。
        *   **Path B (Graph Search)**:
            *   利用提取的实体，在 **Graph Database** 中进行子图遍历（Graph Traversal）。
            *   查找与这些实体直接相连的属性或关系边。
    4.  **Aggregation (聚合)**: 将 Vector 结果（非结构化文本）与 Graph 结果（结构化事实）合并。
    5.  **Re-ranking & Result (重排与输出)**: 对合并后的结果进行去重和相关性打分，返回最终的 `Context` 给用户。

##### 3. Update & Delete (维护操作)
*   **Key Terms (关键术语)**
    *   **Memory ID**: 记忆的唯一标识符（UUID）。
    *   **Re-indexing**: 向量索引重建。

*   **How it works (工作原理 - 像素级流程还原)**
    *   **Delete (删除)**:
        1.  **Lookup**: 系统根据 `memory_id` 查找元数据。
        2.  **Vector Purge**: 从 Vector DB 中物理移除对应的 Embedding 向量。
        3.  **Graph Purge**: 从 Graph DB 中删除对应的节点（如果该节点不再被其他记忆引用）或边。
        4.  **Confirmation**: 返回删除成功信号。
    *   **Update (更新)**:
        本质上是 **Delete + Add** 的原子操作：
        1.  先执行 Delete 流程移除旧版本。
        2.  再执行 Add 流程插入新内容（这会触发新的 Embedding 计算和图谱更新）。

## 3. 工具使用 (Tool Use) - 连接数字世界
LLM 只是一个大脑，工具协议赋予了它手和脚。

### 3.1 Model Context Protocol (MCP)
**核心理念**：**标准化连接**。让所有的 AI 应用都能以统一的方式连接所有的数据源。

#### 深度解剖：它是如何工作的？
在 MCP 出现之前，如果你想让 Agent 读取 Google Drive 文件，你需要查阅 Google 的 API 文档，写专门的 Python 代码。如果你换了 Dropbox，又要重写一遍。

MCP 借鉴了编程界的 LSP（语言服务协议）思想，采用了 **Client-Server** 架构：

1.  **MCP Server (服务端)**：
    *   由数据提供方（如 Google Drive, Slack, 本地文件系统）实现。
    *   它通过标准接口（JSON-RPC）暴露三种能力：
        *   **Resources**：它可以提供什么数据？（类似 GET）
        *   **Tools**：它可以执行什么操作？（类似 POST）
        *   **Prompts**：它有什么预设的提示词？

2.  **MCP Client (客户端)**：
    *   由 Agent 平台（如 Claude Desktop, Cursor, 或你的 LangChain App）实现。
    *   客户端**不需要知道**服务端的具体实现细节，它只需要遵循 MCP 协议与其对话。

3.  *   **价值**：实现了 **O(1)** 的集成复杂度。你写好一个 Agent，就可以连接社区里成百上千个现成的 MCP Server。

#### 核心操作与机制深度解析 (Mechanisms Deep Dive)
MCP 协议定义了一套严格的 JSON-RPC 消息格式，通过标准化的“握手”流程实现连接。

##### 1. Protocol Handshake (协议握手)
在一切开始之前，Client 和 Server 必须先确认眼神。

*   **Key Terms (关键术语)**
    *   **JSON-RPC 2.0**: 底层通信协议。
    *   **Transport**: 传输层（Stdio, SSE, HTTP）。
    *   **Capabilities**: 能力协商（Server 说“我能读文件”，Client 说“我能弹窗”）。

*   **How it works (工作原理 - 像素级流程还原)**
    这是一次标准的握手仪式：
    1.  **Client Hello**: Client 发送 `initialize` 请求。
        *   `{"method": "initialize", "params": {"protocolVersion": "2024-11-05", "capabilities": {...}}}`
    2.  **Server Hello**: Server 返回其支持的功能列表。
        *   `{"result": {"protocolVersion": "2024-11-05", "capabilities": {"resources": {}, "tools": {}}}}`
    3.  **Ack**: Client 发送 `notifications/initialized`。
        *   至此，连接正式建立，双方可以开始互发 Ping/Pong。

##### 2. Resource Subscription (资源订阅)
这不仅仅是读取文件，而是实时监控。

*   **Key Terms (关键术语)**
    *   **URI Scheme**: 统一资源标识符（如 `file:///logs/app.log`）。
    *   **Subscribe**: 订阅指令。
    *   **Notification**: 服务器推送的更新消息。

*   **How it works (工作原理 - 像素级流程还原)**
    1.  **List**: Client 发送 `resources/list`，获取可用资源清单。
    2.  **Subscribe**: Client 发送 `resources/subscribe`，携带 `uri="file:///app.log"`。
    3.  **Watch (监听)**: Server 端启动文件系统监听器（File Watcher）。
    4.  **Push (推送)**: 当文件变动时，Server 主动向 Client 发送 JSON-RPC Notification：
        *   `{"method": "notifications/resources/updated", "params": {"uri": "file:///app.log"}}`
    5.  **Pull (拉取)**: Client 收到通知后，立即发送 `resources/read` 获取最新内容。

##### 3. Tool Execution (工具执行)
这是 Agent 的手。

*   **Key Terms (关键术语)**
    *   **Schema**: 工具的输入参数定义（JSON Schema）。
    *   **Call**: 执行指令。

*   **How it works (工作原理 - 像素级流程还原)**
    这是一次完整的跨进程（甚至跨网络）远程过程调用：
    1.  **Selection (决策)**: Client 端的 LLM 决定调用 `weather_tool`，生成参数 `{"city": "Beijing"}`。
    2.  **Request Construction (封包)**: Client 构造一个 JSON-RPC 请求对象：
        `{"jsonrpc": "2.0", "method": "tools/call", "params": {"name": "weather_tool", "args": {...}}, "id": 1}`
    3.  **Transport (传输)**: 该 JSON 对象被序列化为字符串，通过 Stdio (标准输入输出) 或 HTTP 发送给 Server。
    4.  **Routing (路由)**: Server 收到请求，解析 `method`，将其分发给对应的 Python 函数 `get_weather(city="Beijing")`。
    5.  **Execution (执行)**: 函数运行，查询外部 API，得到结果 "Sunny, 25C"。
    6.  **Response Construction (回包)**: Server 构造响应对象：
        `{"jsonrpc": "2.0", "result": {"content": [{"type": "text", "text": "Sunny, 25C"}]}, "id": 1}`
    7.  **Return (返回)**: 响应传回 Client，Client 提取 `result.content` 并喂给 LLM。

## 4. 知识库工程 (RAG) - 领域认知的构建
RAG (检索增强生成) 是让 Agent 拥有“专业知识”的关键。

### 4.1 GraphRAG (基于图谱的 RAG)
**核心理念**：**见树木，也要见森林**。解决传统 RAG 只能回答局部细节，无法回答全局性问题（Global Sense-making）的缺陷。

#### 深度解剖：它是如何工作的？
想象你有一本百万字的小说。
*   **传统 RAG**：把书切成无数小段。你问“哈利波特在哪一页用了魔法？”，它搜得很准。你问“这本书讲了什么主题？”，它只能随机抓几段，回答得很片面。
*   **GraphRAG**：
    1.  **提取与建图**：LLM 遍历全文，提取出所有人物、地点、事件，构建一个巨大的网络图。
    2.  **社区发现 (Community Detection)**：这是最核心的算法（Leiden 算法）。它会自动把关系紧密的节点聚类成“社区”。比如“凤凰社成员”是一个社区，“食死徒”是另一个社区。
    3.  **层级摘要**：LLM 会为**每个社区**写一段总结。然后把小社区的总结聚在一起，再为上一层的大社区写总结。
    4.  **全局查询**：当你问宏观问题时，GraphRAG 直接使用这些**高层级的社区总结**来回答。这就像是先看目录和章节简介，而不是直接去翻正文。

#### 核心操作与机制深度解析 (Mechanisms Deep Dive)
GraphRAG 的工作流分为“构建期”和“查询期”，其复杂度远超传统 RAG。

##### 1. Indexing (构建索引)
这是一个计算密集型的批处理过程。

*   **Key Terms (关键术语)**
    *   **Chunks**: 文本切片。
    *   **Entity Extraction**: 识别“人、地、物”。
    *   **Community Detection (Leiden 算法)**: 图聚类算法，用于发现紧密联系的子图。
    *   **Community Summaries**: 对每个社区生成的自然语言摘要。

*   **How it works (工作原理 - 像素级流程还原)**
    这是一个极其耗时的**离线批处理流水线**：
    1.  **Text Unitization (切片)**: 输入文档被切分为固定大小的 `TextUnit` (例如 600 token)。
    2.  **Extraction (实体提取)**: LLM 并行处理每个切片，提取实体（Entity）和关系（Relationship）。
        *   *Output*: `(Harry Potter, is_enemy_of, Voldemort)`。
    3.  **Element Summarization (元素描述)**: 如果同一个实体在多个切片中出现，LLM 会汇总生成一段该实体的描述。
    4.  **Graph Construction (建图)**: 使用 NetworkX 构建一个巨大的无向加权图。
    5.  **Community Detection (社区发现)**: 运行 **Leiden 算法**，将图中的节点聚类成层级化的社区（Hierarchical Communities）。
        *   Level 0: 整个霍格沃茨。
        *   Level 1: 格兰芬多学院。
        *   Level 2: 哈利波特的宿舍。
    6.  **Community Summarization (社区摘要)**: **(核心步骤)** LLM 为**每个**社区生成一段总结报告。
        *   *Result*: 系统最终存储的不仅是图，还有成千上万个“微型维基百科条目”。

##### 2. Global Search (全局搜索)
当用户问“这本书的主题是什么？”时。

*   **Key Terms (关键术语)**
    *   **Map-Reduce**: 大数据处理范式。
    *   **Shuffle**: 数据分发。

*   **How it works (工作原理 - 像素级流程还原)**
    这是一个宏大的 Map-Reduce 过程：
    1.  **Analysis (审题)**: LLM 分析用户 Query，确定需要调用的社区层级（例如 Level 2 社区）。
    2.  **Map Step (分发)**:
        *   系统选中该层级下的所有 **Community Summaries**。
        *   将 Summaries 分成多个 Batches（因为 LLM 上下文有限）。
        *   并行询问 LLM：“基于这些摘要，回答用户问题，并给出 0-100 的置信度评分”。
    3.  **Intermediate Answers (中间结果)**: 得到几十个局部答案（R1, R2, R3...）。
    4.  **Reduce Step (归约)**:
        *   过滤掉置信度低（Score < 60）的答案。
        *   将剩下的高分答案拼接，再次送入 LLM。
        *   生成最终的**全局综述**。

##### 3. Local Search (局部搜索)
当用户问“谁杀死了哈利波特？”时。

*   **Key Terms (关键术语)**
    *   **Ego Graph**: 以某个节点为中心的子图。
    *   **Multi-hop Reasoning**: 多跳推理。

*   **How it works (工作原理)**
    1.  **Entity Mapping**: 识别 Query 中的实体（"Harry Potter"）。
    2.  **Expansion**: 找到该实体在图中的邻居（Friends, Enemies），构建 Ego Graph。
    3.  **Traversal**: 沿着边（Relationship）进行有限深度的遍历，检索相关的原始文本块（Chunks）。
    4.  **Generation**: 结合图谱结构信息和原始文本，生成精准答案。

## 5. 评估 (Evaluation) - 质量的度量衡
你怎么知道 Agent 写得好不好？靠感觉是不行的，我们需要工程化的度量标准。

### 5.1 DeepEval (LLM 的单元测试)
**核心理念**：**像测试代码一样测试 AI**。将评估流程集成到 CI/CD 流水线中。

关于 actual_output 参数 “可选” 与 “必需” 的矛盾，核心结论是：语法定义上可选，主流评估场景下必需，具体可归纳为两点：
语法层面（参数定义）：actual_output 是 LLMTestCase 的可选参数，创建对象时不传不会报错，这是为了兼容「仅评估检索器 / 工具调用等无输出组件」的场景如仅验证检索上下文相关性，无需大语言模型生成输出）；
评估层面（实际使用）：deepeval 中 90% 以上的核心评估指标（如答案相关性、信息忠实度、竞技场评估等），均需依赖 actual_output 评估大语言模型的输出质量，因此在端到端评估 RAG / 聊天机器人等主流场景下，该参数是必需的，不传会导致评估报错。

使用deepeval开展评估的理想流程示意图：
![](./assets/4.png)

#### 深度解剖：G-Eval 算法
DeepEval 的核心不仅仅是简单的关键词匹配，它实现了 **G-Eval** 框架。

1.  **LLM-as-a-Judge (以模评模)**：
    *   我们要评测一个 7B 的小模型写得好不好，可以请 GPT-4 来当裁判。
2.  **CoT (思维链) 评分**：
    *   DeepEval 不会直接让 GPT-4 打分，而是强制它生成一段**推理过程**。
    *   *示例*：“请评估这个回答的相关性。步骤1：分析用户的问题核心。步骤2：检查回答是否覆盖了核心点。步骤3：给出 1-5 分。”
    *   这种方法被证明比直接打分更接近人类专家的判断。

#### 核心操作与机制深度解析 (Mechanisms Deep Dive)
DeepEval 将评估过程工程化，使得 AI 系统的质量可度量、可复现。

##### 1. Define Test Case (定义测试用例)
这是评估的基石。

*   **Key Terms (关键术语)**
    *   **Input**: 用户的原始输入。
    *   **Actual Output**: Agent 的实际回答。
    *   **Retrieval Context**: RAG 检索到的文档片段（这是评估“幻觉”的关键）。
    *   **Expected Output**: （可选）黄金标准答案。

*   **How it works (工作原理)**
    DeepEval 并不关心你的 Agent 是用什么框架写的。它只关心**输入输出对**。你构建一个 `LLMTestCase` 对象，实际上就是打包了一个标准化的数据包，准备发送给“法官模型”。

##### 2. Metric Execution (指标计算流水线)
DeepEval 的核心魔法。

*   **Key Terms (关键术语)**
    *   **LLM-as-a-Judge**: 使用一个更强的模型（如 GPT-4）来评估当前模型的表现。
    *   **G-Eval**: 一种通用的评估算法，基于 CoT（思维链）打分。
    *   **Threshold**: 及格线（0-1 之间的小数）。

*   **How it works (工作原理 - 像素级流程还原)**
    DeepEval 的核心是一个**LLM 辅助的评分状态机**：
    1.  **Template Filling (填空)**: 系统将 `Input`, `Actual Output`, `Retrieval Context` 填入预定义的 **G-Eval Prompt 模板**。
    2.  **Judge Invocation (传唤法官)**: 调用配置好的评价模型（通常是 GPT-4o）。
    3.  **CoT Generation (思维链)**: 强制 LLM 先输出一段 `Reasoning`。
        *   *示例*: "模型回答覆盖了用户的核心问题，但忽略了第二个限制条件..."
    4.  **Score Extraction (打分)**: LLM 最后输出一个分数（1-10 或 0-1）。
    5.  **Parsing (解析)**: DeepEval 使用正则或 JSON Mode 解析该分数。
    6.  **Assertion (断言)**: Python 代码比较 `Score >= Threshold`。如果不满足，抛出 `AssertionError`，并附带 LLM 生成的 `Reasoning` 作为错误日志。

##### 3. Synthetic Data Generation (合成数据生成)
如何无中生有地变出测试集？

*   **How it works (工作原理 - 像素级流程还原)**
    DeepEval / Ragas 使用一种**逆向工程**的方法：
    1.  **Chunking (切片)**: 将你的知识库文档切成小块。
    2.  **Generation (生成问题)**: 
        *   LLM 扮演“出题人”，阅读切片，生成对应的问题 `Query`。
        *   *Prompt*: "请根据这段关于 GraphRAG 的文本，出一个困难的问题。"
    3.  **Ground Truth (生成答案)**: 
        *   LLM 扮演“老师”，根据切片内容，写出标准答案 `Expected Output`。
    4.  **Evolution (进化)**: 
        *   (高级步骤) 对生成的问题进行**变异**。
        *   *Simple*: "GraphRAG 是什么？"
        *   *Reasoning*: "GraphRAG 与传统 RAG 相比，在处理全局问题时有何优势？"
        *   通过多轮进化，生成高复杂度的 Golden Dataset。

##### 4. Hyperparameter Tracking (超参数追踪)
如何科学地调优 Agent？

*   **Key Terms (关键术语)**
    *   **Hyperparameters**: 影响模型表现的配置参数（如 `temperature`, `chunk_size`, `model_version`）。
    *   **Decorator**: Python 装饰器 `@deepeval.log_hyperparameters`，用于注入元数据。

*   **How it works (工作原理 - 像素级流程还原)**
    这是一套**元数据挂载机制**：
    1.  **Registration (注册)**: 当你使用 `@deepeval.log_hyperparameters` 装饰一个函数时，DeepEval 的测试运行器（Runner）会在启动时扫描并注册这个钩子函数。
    2.  **Execution (执行)**: 当你运行 `deepeval test run` 命令时：
        *   Runner 初始化一个新的 **Test Run ID**。
        *   Runner 自动执行被装饰的 `hyperparameters()` 函数。
        *   捕获该函数返回的字典 `{ "temperature": 1, "chunk size": 500 }`。
    3.  **Binding (绑定)**: 将这个字典作为 **Metadata (元数据)** 附加到当前的 Test Run 对象上。它不影响测试代码的逻辑，但会像标签一样贴在这次测试结果上。
    4.  **Upload & Visualization (上传与可视化)**: 
        *   测试结束后，DeepEval Client 将 `Test Results` + `Hyperparameters` 打包成 JSON。
        *   发送给 **Confident AI** 云端平台。
        *   在 Dashboard 上，你可以横向比较：当 `chunk size` 从 500 变为 1000 时，`Answer Relevancy` 分数是涨了还是跌了。

##### 5. Multi-turn Evaluation (多轮对话评估)
不仅仅是单问单答，还要看 Agent 是否“记得住”与“聊得顺”。

*   **1. The Foundation: Conversational Test Case (多轮测试用例)**
    与单轮测试不同，多轮评估的基础是 `ConversationalTestCase`，它是一个**黑盒容器**，封装了用户与 Agent 的完整交互历史。
    *   **Key Class**: `ConversationalTestCase`
    *   **Building Block**: `Turn` (包含 `role`, `content`, `retrieval_context`, `tools_called`)。
    *   **Code Anatomy (代码解剖)**:
        ```python
        # 这是一个完整的“剧本”
        test_case = ConversationalTestCase(
            turns=[
                Turn(role="user", content="我想去北京"),
                Turn(role="assistant", content="请问您几号出发？"),
                Turn(role="user", content="明天")
            ]
        )
        ```

*   **2. Core Metrics (核心指标)**
    有了上面的 `test_case`，我们就可以跑以下指标：
    *   **TurnRelevancyMetric (轮次相关性)**: 
        *   *Definition*: 每一轮回答是否紧扣当前问题及历史上下文？
        *   *Check*: 防止 Agent 答非所问或忽视了前几轮的约束条件。
    *   **KnowledgeRetentionMetric (知识保留度)**:
        *   *Definition*: Agent 是否记住了用户之前提供的信息？
        *   *Check*: 用户第1轮说“我叫小明”，第5轮问“我叫什么”，Agent 必须答对。

##### 6. Simulation Mechanism (多轮对话模拟机制)
如何自动化测试多轮对话？我们需要一个“模拟用户”。

![多轮llm交互架构图](./assets/1.png)

*   **Key Terms (关键术语)**
    *   **ConversationalGolden**: 定义测试场景的规则（如“用户是一个挑剔的买家，目标是砍价”）。
    *   **Simulator**: 扮演用户的 LLM，根据 Golden 规则生成输入。
    *   **Bridge**: 连接 Simulator 与你的 Agent 的桥梁。
    *   **Golden vs Test Case**: 
        *   `Golden` (黄金数据): 只有 `input`。这是**题目**。
        *   `Test Case` (测试用例): 包含 `input` + `actual_output`。这是**答卷**。
        *   *Dataset 只有 input 的原因*: Dataset 是题库，你的 Agent 还没做题呢，当然没有答案。

*   **How it works (工作原理 - 像素级流程还原)**
    这其实是两个 LLM 在互搏：
    1.  **Rule Definition (定规则)**: 使用 `ConversationalGolden` 定义剧本：“测什么场景、成功标准是什么”。
    2.  **Bridge Construction (搭桥梁)**: 定义 `model_callback`，让 Simulator 能够像真实用户一样调用你的 Agent API。
    3.  **Simulation Loop (模拟循环)**: 执行 `simulator.simulate()`：
        *   *Turn 1*: Simulator (User) 发起挑战 -> Agent 回复。
        *   *Turn 2*: Simulator 根据 Agent 的回复，决定下一步说什么 -> Agent 回复。
        *   ... 直到达到结束条件。
    4.  **Test Case Generation (生成用例)**: 这一步将“动态的对话过程”固化为“静态的 `conversational_test_cases`”。
    5.  **Evaluation (打分)**: 最后调用 `evaluate()`，对生成的用例进行指标计算。

    > **关键认知**: Dataset 不是直接用来跑分的，而是用来教 Simulator 怎么演戏的。

##### 7. RAG Evaluation (RAG 专项评估)
没有标准答案，如何评估 RAG？

![rag架构图](./assets/2.png)

*   **Key Terms (关键术语)**
    *   **RAG Triad**: 经典的 RAG 评估三剑客。
    *   **Internal Prompt**: DeepEval 内置的评估提示词。

*   **Core Metrics (核心指标)**
    *   **Contextual Precision (检索精确率)**: 检索到的 Top-K 文档里，有多少是有用的？
    *   **Faithfulness (忠实度)**: 回答的内容是否完全源于检索文档？（防幻觉）。
    *   **Answer Relevancy (答案相关性)**: 回答是否解决了用户的问题？

*   **How it works (工作原理 - 无标准答案评估)**
    DeepEval 巧妙地利用了 LLM 自身的逻辑能力：
    1.  **Input**: 只有 `Query`, `Retrieval Context`, `Actual Output` (无需 `Expected Output`)。
    2.  **Internal Prompting (内置推理)**: DeepEval 调用 LLM (如 GPT-4) 并问它：
        *   *Ask*: "请判断 `Actual Output` 中的每一句话，是否都能在 `Retrieval Context` 中找到依据？"
    3.  **Scoring**: LLM 输出 "Yes/No" 或分数，DeepEval 将其转化为指标得分。

##### 8. DeepEval Observer (全链路观测)  @observer
不仅要看结果，还要看过程。

*   **Key Terms (关键术语)**
    *   **Trace**: 一次完整的请求链路。
    *   **Span**: 链路中的一个环节（如“检索”、“生成”）。

*   **Mechanisms (核心能力)**
    *   **基础监控**: 记录耗时、Token 消耗、异常堆栈。
    *   **组件级评估**: 给单个函数（如 `retrieve_nodes`）绑定指标，单独计算检索质量。
    *   **全链路溯源**: 串联组件输入输出，当 Agent 答错时，能看清是检索错了，还是 LLM 理解错了。

##### 9. MCP Evaluation (工具调用评估)
Agent 是否正确使用了工具？

![mcp架构图](./assets/image.png)
![mcp调用evalute](./assets/3.png)

*   **Key Terms (关键术语)**
    *   **MCP Servers**: 可用工具库（告诉 Agent 有什么）。
    *   **MCP Tools Called**: 实际调用记录（告诉 Agent 用了什么）。
    *   **MCPUseMetric**: 专门评估工具调用的指标。

*   **How it works (工作原理 - 语义匹配)**
    这也是一个无需标准答案的过程：
    1.  **Context Assembly (上下文组装)**: DeepEval 收集三要素：
        *   用户问题 ("查一下明天去北京的票")。
        *   可用工具清单 (`[search_ticket, refund_ticket]`)。
        *   实际调用 (`call search_ticket(date='tomorrow', dest='Beijing')`)。
    2.  **Semantic Judgment (语义判断)**: LLM 裁判从三个维度打分：
        *   **Correctness (对不对)**: 是否用了查票工具，而不是退票工具？
        *   **Necessity (必不必要)**: 是否有多余的调用？
        *   **Completeness (完不完整)**: 如果用户问“票价和天气”，是否两个工具都调了？
    3.  **Scoring**: 生成 0-1 的量化分数。

##### 10. LLM Arena (模型竞技场)
*   **Definition**: 在同一套测试集上，让两个不同的 LLM (如 GPT-4 vs Claude-3) 进行 PK，看谁的胜率更高。

### 5.2 Ragas (RAG 专用评估)
**核心理念**：**分而治之**。把 RAG 系统拆解开来分别评估。

#### 深度解剖：RAG 三元组与合成数据
1.  **RAG Triad (三元组指标)**：
    *   **Faithfulness (忠实度)**：回答是否忠实于检索到的文档？（检测幻觉）
    *   **Answer Relevance (相关性)**：回答是否解决了用户的问题？（检测答非所问）
    *   **Context Precision (精度)**：检索到的文档里，有用信息的占比是多少？（检测检索器质量）

2.  **合成数据生成 (Testset Generator)**：
    *   这是 Ragas 最具创新性的功能。在企业里，我们往往只有一堆文档，没有“问题-答案”对用来测试。
    *   Ragas 利用 LLM 的逆向推理能力，**读取你的文档，自动生成**各种难度的问题（简单事实问答、多跳推理问答）和标准答案。这就相当于 AI 自动为你出了一套考试题，用来测试你的 Agent。
