# 中间件智能问答系统技术选型方案 (Production Ready)

针对“中间件智能问答系统”（通常涉及复杂的配置参数、报错代码、专有技术文档），我们需要构建一个**高准确率、低幻觉、可追溯**的 RAG（检索增强生成）系统。

---

## 2. 详细选型理由与对比 (Why & Comparison)

### 2.1 大模型：为什么选 DeepSeek-V3 / Qwen 2.5？
*   **场景特点**：中间件文档包含大量中文技术术语、Java 代码堆栈、XML 配置文件。
*   **DeepSeek-V3 / Qwen 2.5 vs GPT-4o**:
    *   **优势**: 在中文技术领域的理解能力与 GPT-4 持平甚至更优，但成本仅为 GPT-4 的 **1% - 5%**。
    *   **私有化**: 如果中间件文档涉密（如 TongWeb 内部文档），**Qwen 2.5** 是目前开源界最强的模型，支持本地部署，数据不出域。
*   **不选 Llama 3**: 中文支持不如 Qwen 和 DeepSeek 原生好。

### 2.2 编排框架：为什么选 LangGraph？
*   **场景特点**: 用户提问可能是模糊的（如“启动报错了怎么办”）。
*   **LangGraph vs LangChain (Chain)**:
    *   **LangChain Chain (线性)**: 用户提问 -> 检索 -> 回答。如果检索不到，它会强行回答（导致幻觉）。
    *   **LangGraph (图/循环)**: 用户提问 -> 检索 -> **(判断: 文档相关吗?)** ->
        *   *不相关* -> **重写查询** -> 再次检索。
        *   *相关* -> 生成回答。
    *   **结论**: 生产环境必须具备**自我纠错**和**多轮追问**的能力，只有 Graph 架构能优雅实现。

### 2.3 检索策略：为什么必须是混合检索 (Hybrid Search)？
*   **痛点**: 纯向量检索（Vector Search）对**精确匹配**效果很差。
    *   *例子*: 用户搜错误码 `TW-10042`。
    *   *向量检索*: 可能匹配到 `TW-10043`，因为它们在语义上都是“错误码”，向量距离很近。
    *   *关键词检索 (BM25)*: 精确匹配 `TW-10042`。
*   **方案**: 使用 **BM25 (关键词)** + **Dense Vector (语义)** 加权融合（Reciprocal Rank Fusion, RRF）。
*   **不选纯向量**: 在技术文档/运维场景下，参数名和错误码的精确性是致命的，纯向量无法满足。

### 2.4 文档处理 (ETL)：核心难点
*   **场景特点**: 中间件文档里有大量的表格（参数列表）和 XML/YAML 配置。
*   **推荐**: **LlamaParse** 或 **Unstructured**。
*   **理由**: 普通的 PDF 解析器（如 PyPDF2）会把表格读成乱码，导致模型完全看不懂参数对应关系。LlamaParse 专门针对复杂文档结构优化，能保留表格结构。

### 2.5 评估框架：为什么需要 Ragas？
*   **现状**: 你不能直接把系统推给用户，必须先跑通测试。
*   **Ragas**: 提供了 "Context Precision" (检索准不准) 和 "Faithfulness" (回答是不是基于文档) 等指标。
*   **流程**: 在上线前，准备 50 个典型中间件问题（如“如何配置连接池”），跑一遍 Ragas 评分。如果分数低于 0.8，严禁上线。

## 3. 架构设计图 (简述)

```mermaid
graph TD
    User[用户提问] --> Guardrails[安全护栏/意图识别]
    Guardrails -->|闲聊| Chat[直接回复]
    Guardrails -->|技术问题| Rewriter[Query 重写器]
    
    Rewriter --> HybridSearch[混合检索 (ES/Milvus)]
    HybridSearch --> Rerank[重排序 (BGE-Reranker)]
    
    Rerank -->|Top 5 文档| Generator[大模型 (DeepSeek)]
    
    Generator -->|生成答案| HallucinationCheck[幻觉检测]
    HallucinationCheck -->|通过| Output[输出给用户]
    HallucinationCheck -->|不通过| Rewriter
```
