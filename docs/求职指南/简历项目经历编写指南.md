*   **角色**: 核心开发者 / AI 工程师
*   **技术栈**: Python, LangGraph, LangChain, Qwen-Plus (通义千问), DeepEval, mem0 (Vector Memory), RAG (ChromaDB/Qdrant), Pytest

你可以根据简历的篇幅选择 **“精简版”** 或 **“详细版”**。

### 🌟 详细版 (推荐，突出解决问题的能力)

**项目描述**:
构建了一个面向 IT 运维场景的智能体系统，利用 LangGraph 的图编排能力实现复杂的任务规划、多轮对话与自我反思（Reflection）。系统集成了 RAG 知识库检索与长期记忆模块，能够准确处理服务器管理、日志分析及安全合规性检查等任务。

**核心职责与成果**:
*   **架构设计 (Architecture)**: 设计并实现了基于 **LangGraph** 的 StateGraph 架构，包含意图识别、RAG 检索、工具调用、安全审查及结果反思（Reflection）等 5 个核心节点，解决了传统 Chain 结构在复杂任务中死循环的问题。
*   **自动化评估 (Evaluation)**: 搭建了基于 **DeepEval** 的自动化评估流水线，使用 **Qwen-Plus** 作为裁判模型。针对 "幻觉" (Faithfulness) 和 "相关性" (AnswerRelevancy) 建立黄金数据集，**将 Agent 的回答忠实度提升至 100%**，确保运维操作的准确性。
*   **记忆增强 (Memory)**: 集成 **mem0** 向量记忆组件，实现了**用户级 (User Scope) 与项目级 (Project Scope)** 的双层记忆隔离，使 Agent 能够准确记住用户的身份（如“测试工程师”）及偏好，大幅提升交互体验。
*   **工程化落地 (Engineering)**: 编写了自定义的 Pytest 运行器，解决了 Windows 环境下的编码兼容性问题及 LangChain/Pydantic 版本冲突，实现了测试报告的自动化生成与持久化，便于持续集成。
*   **安全合规 (Safety)**: 实现了敏感指令（如读取 `/etc/shadow`）的拦截机制，在 DeepEval 安全评估中达到 **100% 拦截率**，确保企业级数据安全。

### 🚀 精简版 (适合简历空间有限时)

*   **基于 LangGraph 构建智能运维 Agent**: 设计了包含意图识别、检索增强 (RAG)、工具执行和自我反思 (Reflection) 的多节点图结构，实现了复杂的运维任务自动化。
*   **建立 DeepEval 评估体系**: 使用 Qwen-Plus 模型对 Agent 进行自动化评分，覆盖忠实度、相关性等维度，**实现 100% 的安全拦截率和无幻觉回答**。
*   **长期记忆与个性化**: 整合 mem0 向量数据库，实现了跨会话的用户画像记忆（User Scope）和项目上下文记忆（Project Scope）。
*   **解决复杂依赖冲突**: 攻克了 LangChain 与 DeepEval 之间的版本依赖地狱 (Dependency Hell)，通过自定义测试脚本优化了 CI/CD 流程中的日志可读性。

## 3. 面试官可能问到的问题 (准备这些答案)

1.  **Q: 为什么选择 LangGraph 而不是 LangChain 的 AgentExecutor?**
    *   **A**: LangChain 的 AgentExecutor 逻辑比较黑盒，难以精细控制循环和分支。LangGraph 提供了状态机（StateGraph），让我能明确定义“意图识别后是去检索还是直接回答”、“工具报错后是直接结束还是尝试反思重试”，这种确定性对运维场景至关重要。

2.  **Q: 你是如何评估 Agent 的好坏的？**
    *   **A**: 我没有只靠“肉眼看”，而是引入了 DeepEval 框架。我定义了 Golden Dataset（黄金数据集），使用 Qwen-Plus 作为裁判模型，计算 Faithfulness（是否瞎编）和 AnswerRelevancy（是否答非所问）。我还编写了自动化脚本，每次代码提交都会生成 Markdown 评估报告。

3.  **Q: mem0 在项目里具体解决了什么问题？**
    *   **A**: 解决了 LLM“聊完就忘”的问题。通过 mem0，我把用户的身份（如“我是李四”）存入向量库。当用户下次问“我是谁”时，Agent 能从 user_scope 中检索出信息，而不是说“我不知道”。

4.  **Q: 遇到过什么棘手的工程问题？**
    *   **A**: 主要是依赖冲突。DeepEval 依赖较新的 OpenTelemetry，而 google-adk 和 starlette 锁定旧版本。我通过降级非核心库（tenacity, starlette）并编写 `run_eval.py` 脚本来屏蔽 Pydantic 的废弃警告，最终让环境跑通。
