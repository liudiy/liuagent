# 🚀 TongAgent 核心架构与设计白皮书 (Agentic RAG)

本文档事无巨细地记录了 TongAgent 的全貌。经过多次企业级架构重构与能力升级，TongAgent 已经从一个简单的 RAG 检索脚本，蜕变为了对标业界顶尖智能编程助手（如 Claude Code）的企业级 Agentic RAG 系统。

---

## 🌟 核心架构范式：从“传统 RAG”到“Agentic RAG”

TongAgent **彻底废弃了传统的向量数据库（ChromaDB）和静态切片（Chunking）技术**。
传统 RAG 最大的痛点是“上下文断裂”（代码或配置被拦腰截断）和“词频霸权”（宽泛问题找不到微观配置）。

**TongAgent 的解决方案（渐进式揭露架构）：**
将完整的 Markdown/TXT 语料库保留在本地磁盘中。Agent 不再是被动地接受被截断的文本块，而是被赋予了“全套文件系统探索工具”，像一个真实的程序员一样：先看目录 -> 搜索关键字 -> 翻开指定页码阅读上下文。

---

## 🛠️ 核心工具链 (The Toolchain)

为了支撑 Agentic RAG，系统为 Agent 提供了四大核心工具，它们各司其职，形成了完美的“渐进式揭露”闭环：

### 1. `Glob` (全局透视镜)
*   **功能**：使用通配符（如 `**/*TongWeb*.md`）在全局知识库中瞬间定位相关文件。
*   **设计巧思**：它是 Agent 的“无人机视角”。当 Agent 面对海量未知文件时，不需要一层层 `ls` 遍历目录，而是直接用 Glob 俯瞰全局，瞬间找到所有符合命名规则的目标。

### 2. `LS` (目录语义探索器)
*   **功能**：列出指定目录下的所有文件和文件夹。
*   **设计巧思**：弥补正则搜索的“语义缺失”。当 Grep 搜不到具体内容时，Agent 可以通过 `LS` 浏览目录结构，根据文件命名（如 `security_config.md`）进行人类般的“按图索骥”和“语义联想”。

### 3. `Grep` (正则关键字搜索)
*   **功能**：在特定文件或全局搜索特定的关键字，并返回**包含行号**的匹配结果。
*   **设计巧思**：
    *   **强制多词/正则匹配**：Prompt 强制要求 Agent 使用 `SSL|TLS|证书` 这样的正则，而不是单搜 `SSL`，极大地提高了召回率。
    *   **只为定位，不为阅读**：Grep 的主要目的不是为了看内容，而是为了**拿到关键信息所在的行号**，为下一步的阅读做铺垫。

### 4. `Read` (带分页的精准阅读器)
*   **功能**：给定文件路径、起始行号（`start_line`）和结束行号（`end_line`），读取文件内容。
*   **设计巧思（动态切片）**：这是对传统 RAG 的降维打击。工具内部写死了 `end_line - start_line <= 500` 的硬限制。它强制 Agent 像查字典一样一页一页看，既避免了 Token 溢出（Lost in the middle），又保证了上下文 100% 连贯。如果没看全，Agent 会自主决定读取下一页。

## 🧠 独家设计巧思与核心机制 (Secret Sauce)

TongAgent 的强大不仅仅在于工具，更在于其底层的调度逻辑和防呆机制。

### 💡 巧思 1：强制思考拦截器 (Mandatory Thought Interceptor)
*   **痛点**：大模型经常“偷懒”，不经过大脑就直接瞎调工具，导致陷入死循环。
*   **机制**：在 `tool_exec.py` 节点中实现了一个中间件。在真正执行工具前，拦截器会扫描模型的输出，**如果没有发现完整的 `<thought>...</thought>` 标签，拦截器会直接伪造一个工具报错**：`"System Error: TOOL EXECUTION BLOCKED..."`。
*   **效果**：程序不会崩溃！它会强制把这个假错误扔回给 Agent，逼迫 Agent 在 ReAct 循环中重新思考、重新调用。这极大地提升了复杂任务的成功率和意图对齐度。

### 💡 巧思 2：Map-Reduce 多并发规划器 (Parallel Planner)
*   **痛点**：当用户问“请告诉我 THS、TongWeb 和 TongRDS 怎么配置 SSL”时，单线程 Agent 容易查完第一个就忘了后面两个，或者把三者的配置搞混。
*   **机制**：引入 Planner 节点。基于 **MECE 原则（相互独立，完全穷举）**，Planner 会将这个大问题拆解为 3 个平行的子任务（THS任务、TongWeb任务、TongRDS任务）。然后系统会拉起 3 个并行的 Worker 进程同时去探索这三个组件。
*   **效果**：探索速度翻倍，且各组件知识物理隔离，绝不串味。

### 💡 巧思 3：严苛的评估者节点 (Evaluator Review Loop)
*   **机制**：Agent 探索完信息后不能直接输出给用户，必须先过 `Evaluator`（评估节点）这一关。
*   **打分逻辑**：
    *   `approved`（完美）：信息完整且有代码/路径支撑，放行。
    *   `partial_approved`（部分通过）：比如找到了 TongWeb，但 THS 没找全。Evaluator 会写下“批语”（如：请继续寻找 THS 的主配置文件路径），然后**把 Agent 踢回 `intent_parse` 节点继续找**。
    *   `rejected`（打回）：全错，重找。
*   **效果**：这是 Agent 能够给出极高质量、极其详实回答的终极保障。用户看到的永远是经过内部多轮打磨的完美答案。

### 💡 巧思 4：网络波动自愈机制 (Resiliency)
*   **机制**：在 `llm_factory.py` 中，针对大模型 API 经常出现的 TCP 强制断开（`httpcore.RemoteProtocolError`）和超时问题，配置了底层的 `max_retries=3` 和 `timeout=120`。
*   **效果**：当网络极度不稳定时，Agent 会在底层静默重试，而不会把冰冷的报错直接拍在用户脸上，保证了长耗时任务的存活率。

### 💡 巧思 5：长短期记忆融合 (Memory System)
*   **短期工作记忆 (Working Memory)**：在每一次 ReAct 循环中，Agent 会把查到的高价值线索（如某配置的完整段落）用“严格榨取”法则提炼出来，塞进短期记忆池。最终生成答案时，模型只看这个高浓缩的记忆池，而不看几万字的废话聊天记录。
*   **长期核心记忆 (Core Memory)**：系统集成了长效记忆机制。比如用户曾强调过“必须使用虚拟环境运行”或“TongRDS 是缓存不是关系型数据库”，这些知识会跨越会话保留，并在系统初始化时作为 System Prompt 注入，让 Agent “越用越懂你”。

### 💡 巧思 6：Claude Code 风格的 Token 上下文折叠 (Context Folding)
*   **痛点**：在多轮 ReAct 循环中，每次调用 `Read` 或 `Grep` 返回的长文本都会堆积在 `messages` 历史记录中。如果循环 10 次，最后一次对话会将前面 9 次的废话全部发给大模型，导致 Token 消耗呈指数级爆炸，且极易引发大模型“Lost in the Middle（中间遗忘）”综合征。
*   **机制 (Context Pruning)**：
    *   **动态历史修剪 (折叠 ToolMessage)**：在路由节点 (`intent_parse.py`) 中，除了最近的一次 ToolMessage 保留部分内容外，所有**历史轮次的 ToolMessage 都会被强制清空/折叠**。
    *   **折叠思考过程 (折叠 AIMessage)**：既然旧的工具返回结果已被折叠，那么 AI 针对旧结果长篇大论的 `<thought>` 思考过程也没有保留价值。系统会同步将历史的 AIMessage 折叠为 `[历史思考过程已折叠]`，仅保留其发出的 `tool_calls` 意图以维持逻辑连贯性。
*   **效果**：完美复刻了 Claude Code 和 Cursor 的 Token 控制策略。由于有“工作记忆（Working Memory）”作为兜底，Agent 绝不会忘记关键信息，但每次对话发给 API 的 Token 数被暴降了 80% 以上，极大地降低了运行成本，同时提升了响应速度和推理智商。

### 💡 巧思 7：Prompt Caching (提示词缓存架构预埋)
*   **痛点**：为了让 Agent 足够聪明，我们给它注入了长达几千字的 System Prompt（包含所有工具规范、XML 标签规则等），这部分内容在每一次 ReAct 循环中都会被重复发送。
*   **机制**：在架构底层，我们已经为 `sys_msg` 预留了 **Prompt Caching** (如 Anthropic 提供的 `{"cache_control": {"type": "ephemeral"}}` 机制)。
*   **效果**：一旦接入原生支持缓存的模型 API，系统提示词将仅在第一次请求时收费，后续所有轮次读取均享受缓存折扣。这使得“塞入大量规范甚至整本开发手册作为系统提示词”成为了极其低成本的操作。

## 📂 企业级工程目录规范

目前的目录结构严格遵循了现代软件工程（如 DDD 领域驱动设计）的理念：

*   `tong_agent/core/`: 核心大脑（状态管理、模型工厂、工具注册表）
*   `tong_agent/nodes/`: 动作节点（规划、意图解析、工具执行、评估、回答生成）
*   `tong_agent/tools/`: 手脚（文件读写、搜索、Glob等）
*   `tong_agent/prompts/`: 灵魂（所有 System Prompts 均从代码中剥离，以 Markdown 形式维护，极易调优）
*   `tong_agent/config/`: 神经（环境变量、日志配置等统一管理）
