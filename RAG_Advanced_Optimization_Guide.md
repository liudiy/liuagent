# 🚀 RAG 进阶优化全景指南 (Advanced RAG Optimization Guide)

在构建中间件领域的专业问答系统时，RAG（检索增强生成）的准确率和召回率决定了系统的成败。为了避免频繁地重新执行耗时的 Chunking（分块）和 Embedding（向量化）过程，我们应当**在建库期一次性做重资产的结构化处理**，随后将主要的优化精力集中在**检索期（Retrieval-time）**和**生成期（Generation-time）**的轻量级策略上。

本指南梳理了未来可用于深度优化 RAG 系统的所有核心技术方案。

---

## 🏗️ 第一阶段：建库期优化 (Indexing-time)
*注：建库期优化属于“重资产”操作，每次修改都需要重新跑模型生成向量。建议将以下策略一次性实现在预处理脚本中，生成最终版向量库后不再轻易改动。*

### 1. 结构化切分 (Structural Chunking)
*   **痛点**：传统的按固定字数（如 `RecursiveCharacterTextSplitter` 设为 500 字）切分，容易将完整的代码块、表格或上下文逻辑腰斩。
*   **方案**：
    *   **Markdown-based Chunking**：利用 `MarkdownHeaderTextSplitter` 按照文档的 `#`, `##` 标题层级进行切分。
    *   **语义切分 (Semantic Chunking)**：利用小模型或 NLTK 识别段落语义边界，确保每个 Chunk 都是一个完整的知识单元（例如，一个报错的完整排查步骤）。

### 2. 父子文档检索架构 (Parent-Child Retriever)
*   **痛点**：Chunk 设得太大（如 2000 字），向量匹配不准；设得太小（如 200 字），大模型回答时缺乏上下文。
*   **方案**：使用 LangChain 的 `ParentDocumentRetriever`。
    *   **存储逻辑**：将大块文档（父文档）存入内存或 Redis 键值库；将其进一步切分为小块（子文档），存入 Chroma 向量库，并在元数据中绑定父文档的 ID。
    *   **检索逻辑**：检索时通过小块进行高精度匹配，但最终提取其对应的父文档喂给大模型。

### 3. 假设性问题注入 (Hypothetical Questions / Query2Doc)
*   **痛点**：用户提问（口语化、简短）与官方文档（书面化、冗长）之间存在语义鸿沟。
*   **方案**：在切分出一个 Chunk 后，调用轻量级 LLM（如 Qwen-Turbo）为其生成 3~5 个“用户可能会怎么问”的假设性问题，并将这些问题与原文合并或独立 Embedding 映射到该 Chunk。

### 4. 丰富的元数据提取 (Rich Metadata Extraction)
*   **方案**：入库时，通过规则或 LLM 提取文档的属性（如：`source`, `version`, `category: config/error`, `chapter`）并存入 `metadata`。为后续的“元数据前置过滤（Pre-filtering）”打下基础。

## 🔍 第二阶段：检索期优化 (Retrieval-time)
*注：这一阶段的优化纯粹是代码逻辑层面的调整，不需要重新构建向量库。*

### 1. 引入重排层 (Cross-Encoder Reranker)
*   **痛点**：Vector 相似度和 BM25 只能匹配字面或浅层语义，常常召回“包含关键词但不相关”的噪音。
*   **方案**：
    *   放宽初筛：通过 Vector + BM25 混合检索一口气召回 20~30 个候选文档。
    *   精准重排：引入 `BAAI/bge-reranker-v2-m3` 等交叉编码器模型，将 Query 和这 30 个文档逐一组合打分。
    *   截断：只保留真实语义匹配度最高的 Top-5。

### 2. 查询路由与多路分发 (Query Routing)
*   **方案**：在执行检索前，先用轻量级模型或规则判断用户的意图。
    *   **精准检索流**：如果询问具体报错，走常规 Vector + BM25 路线。
    *   **全局总结流**：如果询问“TongWeb 的整体架构”，则绕过向量库，去查全局的知识图谱（GraphRAG）或专门存放摘要的集合。
    *   **结构化查询**：如果是对比问题，将 Query 转换为 SQL 查询结构化数据库。

### 3. 查询拆解 (Query Decomposition)
*   **痛点**：用户抛出复杂问题：“TongWeb 怎么安装？它和 Tomcat 在配置连接池上有什么区别？”单次检索极难命中。
*   **方案**：利用 LLM 将复杂问题拆解为多个单一的子问题（Sub-queries），并发执行多次独立检索，最后将召回的上下文合并。

### 4. 元数据前置过滤 (Pre-filtering / Self-Querying)
*   **方案**：结合建库期的元数据，在检索前让大模型将自然语言转化为带有 Filter 的查询。例如用户问“TongWeb 7.0 怎么配置”，系统自动将过滤条件设置为 `filter={"version": "7.0"}`，缩小搜索范围。

## ✍️ 第三阶段：生成期优化 (Generation-time)
*注：发生在拿到检索结果后，喂给大模型生成最终回答之前的阶段。主要目的是防幻觉和降本增效。*

### 1. 上下文压缩与过滤 (Prompt Compression / Context Filter)
*   **痛点**：即便经过 Rerank，Top-5 文档中依然可能存在无关的“废话”，占用 Token 并干扰大模型。
*   **方案**：使用 `LLMLingua` 或让小模型快速过一遍文档，将无关的段落剔除，只保留核心实体和上下文。

### 2. 自步进检索与批判 (Self-RAG / Active Retrieval)
*   **方案**：在 LangGraph 架构中引入一个 **评判节点 (Grader Node)**。
    *   拿到检索结果后，先不急着生成，而是让 LLM 评估：“这些文档包含回答用户问题所需的信息吗？”
    *   如果包含：进入生成节点。
    *   如果不包含：触发重试，让 LLM 重写查询关键词（Query Rewrite），再次调用检索工具。

### 3. 严格的引用约束 (Citation / Grounding)
*   **方案**：在最终的 System Prompt 中，强制大模型进行来源标注。
    *   指令示例：“你必须且只能基于以下提供的文档进行回答。在你的回答中，每一句话的末尾都必须使用 `[来源：文件名]` 标注出处。如果文档中没有相关信息，请直接回答‘知识库中未找到相关内容’，绝不允许编造。”

---

## 🎯 实施路径建议
为了达到最高的投入产出比，建议按以下顺序推进：
1. **优先落地**：【检索期】的 **Reranker 重排序** 和 **Query 拆解/HyDE**（立竿见影，无需动库）。
2. **阵痛重构**：一次性重构数据预处理脚本，实现【建库期】的 **父子文档 (Parent-Child)** 和 **结构化切分**，生成最终版 V3 向量库。
3. **架构升级**：在 LangGraph 中实现【生成期】的 **Self-RAG (检索结果自评判循环)**。
